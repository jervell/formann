# Triage

Move issues on the project issue tracker through a small state machine of triage roles.

Every comment or issue posted by `/triage` **must** start with this disclaimer:

```
> *This was generated by AI during triage.*
```

## Reference docs

- `${CLAUDE_SKILL_DIR}/AGENT-BRIEF.md` — how to write durable agent briefs
- `${CLAUDE_SKILL_DIR}/OUT-OF-SCOPE.md` — how the `.out-of-scope/` knowledge base works
- `docs/formann/issue-tracker/BINDING.md` — how tracker operations are realized in this project; read before acting

## Triage state

Every triaged issue is tagged with a state, a category, and a type.

State transitions: an untriaged issue is first set to `needs-triage`; from there it transitions to `needs-info`, `ready-for-agent`, `ready-for-human`, or `wontfix`. `needs-info` returns to `needs-triage` once the open questions are answered. The maintainer can override at any time — flag transitions that look unusual and ask before proceeding.

## Triage's job

Triage exists to resolve open questions so the issue can leave with a decided type and state. Apply this decision tree at the end of triage:

1. **Are open questions remaining?** If yes, resolve them now in this session — read code, grill the maintainer, decide. Don't park the issue with open questions by tagging it HITL or `ready-for-human`; those tags describe execution mode and ownership, not "triage isn't finished".
2. **Can an agent meaningfully start this work?** If yes → state `ready-for-agent` (the default). If no — external access required, manual production step, real-time human judgment — → state `ready-for-human`.
3. **During execution, will the work need human-in-loop gates?** Design pivot points, review-before-merge on something irreversible, etc. If yes → type `HITL`. If no → type `AFK` (the default).

`AFK + ready-for-agent` is the common exit. `HITL + ready-for-agent` is also common — agent runs the work, gates with the maintainer at decision points. `ready-for-human` is rare.

## Invocation

The maintainer invokes `/triage` and describes what they want in natural language. Interpret the request and act. Examples:

- "Show me anything that needs my attention"
- "Let's look at #42"
- "Move #42 to ready-for-agent"
- "What's ready for agents to pick up?"

## Show what needs attention

Query the issue tracker and present four buckets, oldest first:

1. **Untriaged** — never triaged.
2. **`needs-triage`** — evaluation in progress.
3. **`needs-info` with new activity since the last triage notes** — needs re-evaluation.
4. **`in-review`** — implementer has shipped, awaiting maintainer signoff.

Show counts and a one-line summary per issue. Let the maintainer pick.

## Triage a specific issue

1. **Gather context.** Read the full issue (body, comments, labels, state) per the **Read the issue** verb in `docs/formann/issue-tracker/BINDING.md`. Parse any prior triage notes so you don't re-ask resolved questions. Explore the codebase using the project's domain glossary, respecting ADRs in the area. Read `.out-of-scope/*.md` and surface any prior rejection that resembles this issue.

2. **Recommend.** Tell the maintainer your category, state, and type recommendation with reasoning, plus a brief codebase summary relevant to the issue. Wait for direction.

3. **Reproduce (bugs only).** Before any grilling, attempt reproduction: read the reproduction steps, trace the relevant code, run tests or commands. Report what happened — successful repro with code path, failed repro, or insufficient detail (a strong `needs-info` signal). A confirmed repro makes a much stronger agent brief.

4. **Grill.** If the issue body has open design questions, undecided options, or "TBD during triage" markers, run a `/grill-with-docs` session. Don't substitute inline clarifying questions — that's unstructured grilling, and it misses branches.

5. **Apply the outcome.** Before transitioning to `ready-for-agent` or `ready-for-human`, invoke the **Make issue runner-ready** verb. If it refuses (binding-specific reasons, typically a missing slug under GH-issues), prompt the maintainer for the missing input and re-invoke. If unresolvable, refuse the state transition — the issue stays in its prior state (typically `needs-triage`, or `needs-info` if open questions remain).

   Set the issue's state to the chosen value, plus the side effects below:
   - `ready-for-agent` — Publish the agent brief (see `${CLAUDE_SKILL_DIR}/AGENT-BRIEF.md`). If the issue is type `HITL`, the brief should call out the gates the agent must check in at.
   - `ready-for-human` — Publish the agent brief with a note on why an agent can't start (external access, manual production step, etc.).
   - `needs-info` — Record triage notes (template below).
   - `wontfix` (bug) — Comment with Wontfix explanation.
   - `wontfix` (enhancement) — write to `.out-of-scope/` (see `${CLAUDE_SKILL_DIR}/OUT-OF-SCOPE.md`); Comment with Wontfix explanation linking to the out-of-scope file.
   - `done` — accepting an `in-review` issue. Optionally Comment with Note if there's something worth recording.
   - `ready-for-agent` (from `in-review`) — rejecting; Comment with Rework notes explaining what needs to change.
   - `needs-triage` — optionally Comment with Note if there's partial progress.

## Verify an `in-review` issue

When accepting or rejecting an `in-review` issue:

1. **Read the latest Implementation comment's Evidence block** (later ones supersede earlier on reworked issues). Trust automated coverage by default: a criterion pointing at named tests with the build green, or at quoted command output, is verified. Each `[x]` is the agent's claim, not your verdict — spot-check the proof for any tick you doubt.
2. **Walk the `[human]` rows.** Each `[ ]` row in the Evidence block carries a one-line ask. Drive the UI, hit the endpoint, read the prose — whatever the row calls for. Decide pass or fail per row.
3. **Record the walk.** Comment with Verification on the issue listing each walked row and the maintainer's verdict (see template below). Skipped on rejection — the rework's new Implementation comment supersedes.
4. **Decide.**
   - Accept → state `done`.
   - Reject → state `ready-for-agent` with rework notes naming what failed and how.

If the Evidence block is missing or doesn't map to the criteria, reject with rework notes asking for one. The next `/implement` run produces a fresh Evidence block and re-ships.

State `done` is the per-issue signoff. For features run via the AFK runner — where the gate auto-`done`s clean issues without walking `[human]` rows — the feature-level final human gate happens at Archive (see below). Archive walks any `[ ]` rows that didn't get walked at Verify.

### Verification comment template

```markdown
### Verification — YYYY-MM-DD

> *This was generated by AI during triage.*

**Walked**

- **PASS:** "Layout reads naturally…" — drove `/voice/chat`, sent a voice message; waveform renders, audio plays.
- **FAIL:** "Copy is concise…" — body text too verbose; trim to 2 sentences.
```

Quote the criterion's leading text using the same rules as the Evidence block (3–8 words, ending in `…`, unique within the brief). One bullet per walked row. Failure bullets carry a one-line reason that's actionable for rework.

## Quick state override

If the maintainer says "move #42 to ready-for-agent" (or "to done", etc.), trust them and set the state directly. Confirm what you're about to do — the state transition and any side effect step 5 mandates for the target state — then act. Skip grilling. If moving to `ready-for-agent` and the issue has no existing agent brief, ask whether they want to grill it out or publish one directly.

## Archive a completed feature

When every issue in a feature is terminal (`done` or `wontfix`) and the feature is shipped, archive the feature.

Trigger: the maintainer says "archive `<feature>`" or similar. Don't auto-archive — wait for the explicit request.

Archive is the feature-level final human gate. Even when the AFK runner has auto-`done`'d issues without walking `[human]` rows, those rows still need walking before the feature ships. The ritual is structured so nothing slips through.

1. **Precheck.** Confirm every issue is terminal (`done` / `wontfix`). If anything is in flight, refuse.
2. **Enumerate pending `[human]` rows.** For each `done` issue, read the latest Implementation comment's Evidence block and collect every `[ ]` row whose criterion is **not** already marked PASS in a subsequent Verification comment on the same issue. Skip `wontfix` issues. The result is a per-issue checklist of unwalked `[human]` items.
3. **Walk with the maintainer.** Present the consolidated checklist, grouped by issue. The maintainer drives the necessary checks and replies with verdicts: pass per row, or fail with a reason.
4. **Record the walk.** For each issue with walked rows, Comment with Verification listing the rows and verdicts (PASS / FAIL — same template as the Verify step).
5. **Handle failures.** If any row failed, that issue moves back to `ready-for-agent` with rework notes naming the failed rows. Archive aborts. Rework ships via `/implement`; return to Archive afterward.
6. **Archive the feature.** Per the binding's archive convention.

If step 2 yields an empty checklist (every Evidence block is all `[x]`, or every `[ ]` row already has a PASS verdict in a Verification comment), skip steps 3–5 and proceed to step 6.

## Needs-info template

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**Open questions:**

- question 1
- question 2
```

Capture everything resolved during grilling under "established so far" so the work isn't lost. Questions must be specific and actionable, not "please provide more info".

## Resuming a previous session

If prior triage notes exist on the issue, read them, check whether the open questions have been answered, and present an updated picture before continuing. Don't re-ask resolved questions.
