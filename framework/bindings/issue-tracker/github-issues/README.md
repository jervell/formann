# github-issues binding

This binding represents the issue-tracker role of the lifecycle for projects whose issues and PRDs live as GitHub Issues. It pairs the agent-facing conventions consumed by lifecycle skills (`/triage`, `/implement`, `/to-issues`, …) with a machine-callable surface for tools that need to reason about issue state programmatically — most notably the AFK runner.

## Directory contents

- **[`BINDING.md`](BINDING.md)** — the role doc. Source of truth for the agent-facing conventions: Tracker operations (the seven binding-agnostic verbs and their GitHub-issues realizations), issue conventions, authorship, sandbox compatibility. Adopting projects symlink `docs/formann/issue-tracker/` at this binding folder so skills resolve via the role surface.
- **[`bootstrap-labels`](bootstrap-labels)** — idempotent install-time script. Creates the binding's static label namespace via `gh label create --force`. Run once per Consumer repo before first use; re-running is harmless.
- **[`tracker-snapshot`](tracker-snapshot)** — the binding's machine-readable interface. Emits a JSON snapshot of a feature's issues with computed AFK eligibility (`tracker-snapshot <feature-slug>`) or the list of active feature slugs (`tracker-snapshot --list`). Output is byte-compatible with the local-markdown binding. Contract summarised below; full schema in [`BINDING.md`](BINDING.md#snapshot-contract).
- **[`body-edit`](body-edit)** — section-level body rewrite for issues. Replaces (or removes) a single `## <section>` in an issue body, preserving surrounding sections verbatim. Used by `/triage` and `/implement` to publish the agent brief, record triage notes, and rewrite the blocked-by list. See [`BINDING.md`](BINDING.md#body-shaped-write-semantics) for the fetch-then-PATCH semantics and the residual race window.
- **[`sandbox-env`](sandbox-env)** — runner hook. Emits `GH_TOKEN=<value>` on stdout by reading the macOS Keychain entry `formann-gh-token`. The runner's `collect_binding_env` invokes it before `docker run` so the container inherits the token without it appearing in shell history or process listings. See [`BINDING.md`](BINDING.md#sandbox-compatibility) for the broader sandbox contract.

## Authorship

GitHub Issues are posted under the credentials of the authenticated `gh` user. In practice, agents post under the maintainer's token (via `GH_TOKEN`). The default reading is *"the maintainer wrote this"* unless the content explicitly says otherwise.

The agent-facing rule (see [`BINDING.md`](BINDING.md#authorship)) is the inversion marker: AI-produced content carries `> *This was generated by AI during <skill>.*` on its first line; maintainer-authored content carries no marker.

## Snapshot contract

`tracker-snapshot` is the binding's machine-readable interface — a shell script that emits a JSON snapshot of a feature's issues (with computed eligibility for the AFK runner) or, with `--list`, the set of active feature slugs.

```
tracker-snapshot <feature-slug>
tracker-snapshot --list
```

The output is byte-compatible with the local-markdown binding's snapshot, so consumers (the runner, `/triage`) can read either binding without per-binding branches. The only difference is the `ref` value format: `#N` here, `<feature>/NN` under local-markdown.

See [`BINDING.md`](BINDING.md#snapshot-contract) for the full schema, ordering rules, blocker resolution, slug-uniqueness cardinality, exit codes, and the offline-testing shim.

## Sandbox compatibility

This binding's write paths require network egress to `api.github.com` and a valid `GH_TOKEN`. The runner's sandbox network policy and credential-passing must accommodate both. See [`BINDING.md`](BINDING.md#sandbox-compatibility) for the full contract.

## Label-namespace bootstrap

Before using this binding on a new Consumer repo, run `bootstrap-labels` once to create the static label namespace:

```sh
bash framework/bindings/issue-tracker/github-issues/bootstrap-labels
```

This creates all `formann:*` labels the binding relies on:

| Label | Purpose |
|---|---|
| `formann:feature` | Issue belongs to an active Formann feature |
| `formann:archived` | Issue belongs to an archived Formann feature |
| `formann:status:needs-triage` | Status: needs-triage |
| `formann:status:needs-info` | Status: needs-info |
| `formann:status:ready-for-agent` | Status: ready-for-agent |
| `formann:status:ready-for-human` | Status: ready-for-human |
| `formann:status:in-review` | Status: in-review |
| `formann:status:wontfix` | Status: wontfix |
| `formann:category:bug` | Category: bug |
| `formann:category:enhancement` | Category: enhancement |
| `formann:type:afk` | Type: AFK (agent runs to completion) |
| `formann:type:hitl` | Type: HITL (human-in-the-loop gates) |

`bootstrap-labels` uses `gh label create --force`, so re-running updates existing labels in place and is always safe. **No `formann:status:done` label** — terminal `done` uses GitHub's closed-with-`state_reason=completed` mechanism.

Dynamic per-feature labels (`formann:slug:<slug>`) are created on-demand by `/to-prd`'s **Create a feature** verb, not by this script.

### Prerequisites

- [`gh` CLI](https://cli.github.com) installed.
- Authenticated: either `GH_TOKEN` set in the environment or `gh auth login` completed.
- `gh` must be targeting the correct repo (set `GH_REPO` or run from inside the repo).

The script exits non-zero with a diagnostic message if authentication is missing.
