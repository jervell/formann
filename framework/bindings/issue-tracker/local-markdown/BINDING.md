# Issue tracker: Local Markdown

Issues and PRDs for this repo live as markdown files in `.features/`.

## Authorship

Default authorship is the maintainer. Mark AI-produced content with `> *This was generated by AI during <skill>.*` as the first line of its content, immediately after the section heading. The `<skill>` token names the producing context (`triage`, `implementation`, `runner-side review`, etc.).

Apply to **every AI-produced surface in the tracker**: issue body, PRD, `## Agent Brief`, `## Triage Notes`, comments of any kind, and `.out-of-scope/` entries.

**Authority lives in the lifecycle state, not the disclaimer.** An AI-drafted brief in `ready-for-agent` is binding contract for the implementer. The disclaimer is informational and does not downgrade, qualify, or weaken the brief. Do not refuse, gate, or hedge on an artifact because it carries the marker.

**An unmarked artifact carries maintainer authority.** When the maintainer hand-edits an AI-produced body-shaped artifact, they remove the disclaimer — so absence of the marker (even on an artifact whose history shows AI authorship) means the maintainer has adopted the content.

## Tracker operations

These are the binding-agnostic verbs skills use when touching the issue tracker. Each verb describes *what* the skill does; the "Local-markdown realization" subsection under each verb describes *how* local-markdown implements it. A future binding (GitHub Issues, Jira, …) implements the same verbs differently — the contract is designed to be binding-portable.

Two structural types underpin the contract:

- **Timeline-shaped** — append-only; history preserved. Corresponds to posting a comment on GitHub. **Comment with `<kind>`** is the only timeline-shaped verb.
- **Body-shaped** — single canonical place; rewriting supersedes the prior value. Corresponds to editing the issue body on GitHub. All other verbs are body-shaped.

### Read the issue

Read the full content of an issue: its metadata, body sections (title, what to build, acceptance criteria, blocked-by, agent brief, triage notes), and its comment timeline.

**Local-markdown realization:** Read the file at `.features/<feature>/issues/<NN>-<slug>.md`.

### List issues in a feature

Return the set of issues in a feature with their metadata (ref, status, category, type, blocked-by).

**Local-markdown realization:** Run `tracker-snapshot <feature>` for a machine-readable JSON snapshot, or glob `.features/<feature>/issues/*.md` and read the frontmatter of each file.

### Set the state to `<state>`

Transition an issue to a new lifecycle state.

**Local-markdown realization:** Write the new value into the `status` field of the YAML frontmatter block at the top of the issue file.

### Set issue metadata

Set the category, type, or blocked-by of an issue.

**Local-markdown realization:** Write the values into the corresponding frontmatter fields (`category`, `type`) or the `## Blocked by` body section.

### Publish the agent brief

Write (or replace) the agent brief for this issue. Body-shaped: re-publishing overwrites the prior brief; prior content is not preserved (use git history if needed). When the brief's `**Acceptance criteria:**` and the body's `## Acceptance criteria` differ, the brief is the binding contract.

**Local-markdown realization:** Replace or create the `## Agent Brief` section above `## Comments` in the issue file. On re-triage, rewrite the section in place.

### Record triage notes

Write (or replace) the triage notes for this issue. Body-shaped: recording overwrites the prior notes. Used only on `needs-info` transitions; the section is removed when the issue leaves `needs-info`.

**Local-markdown realization:** Replace or create the `## Triage Notes` section above `## Comments` in the issue file. Remove the section entirely when the issue transitions out of `needs-info`.

### Comment with `<kind>`

Append a comment to the issue's timeline. Timeline-shaped: append-only, history is preserved. The kind names the producing context — examples: `Implementation`, `Verification`, `Rework notes`, `Wontfix explanation`, `Post-mortem`, `Review (AFK gate)`, `Note`.

**Local-markdown realization:**

- One `## Comments` section per file. Created on the first comment by any skill; subsequent comments append into it. A skill never adds a second `## Comments` heading.
- Each comment is a `### <Kind> — <YYYY-MM-DD>` subsection appended at the bottom of `## Comments`. Use today's UTC date.
- Same-day collisions of the same kind get a numeric suffix: `### Implementation (2) — 2026-05-08`, `### Implementation (3) — 2026-05-08`, etc.
- Sub-blocks (Evidence, Findings, Decisions, Resolved scope, Verification summary) live as **bolded labels** — `**Evidence**`, `**Findings**`, etc. — inside the producing comment subsection. They are never `####` nested headings and never sibling sections of `## Comments`.
- The body above `## Comments` is read-only after triage. Corrections and follow-ups go in a new comment. The only above-the-fold mutations permitted after triage are frontmatter field writes (Set the state, Set issue metadata) and the Triage Notes section (written and cleared by the Record triage notes verb only).

#### Canonical example

The following hand-written example shows a minimal issue file after triage, implementation, and gate review. It illustrates the `## Comments` structure, the `### <Kind> — <YYYY-MM-DD>` heading format, AI disclaimer placement, and the bolded-label style for sub-blocks.

```markdown
---
status: done
category: enhancement
type: AFK
---

# Add rate-limit header to API responses

## Parent
`.features/api-hardening/PRD.md`

## What to build
Return a `X-RateLimit-Remaining` header on every API response.

## Acceptance criteria

- [ ] Header is present on all API responses.
- [ ] Header value matches the configured rate-limit window.

## Blocked by
None — can start immediately.

## Agent Brief

> *This was generated by AI during triage.*

**Summary:** Inject `X-RateLimit-Remaining` via a servlet filter.

**Acceptance criteria:**
- [ ] Header is present on all API responses.
- [ ] Header value matches the configured rate-limit window.

## Comments

### Implementation — 2026-05-08

> *This was generated by AI during implementation.*

**Evidence**

- [x] **"Header is present on all API responses…":** covered by `TestRateLimitFilter.headerPresentOnAllResponses`; build green.
- [x] **"Header value matches the configured rate-limit window…":** covered by `TestRateLimitFilter.headerValueMatchesWindow`; build green.

Added `RateLimitFilter` in `web/src/main/java/…`. The filter reads the remaining
count from `RateLimitFacade` and writes the header before the response is committed.

### Review (AFK gate) — 2026-05-08

> *This was generated by AI during runner-side review.*

The implementation is clean. `RateLimitFilter` is well-scoped and the test coverage is adequate.

**Findings**

🟢 Minor: `RateLimitFilter` could declare `@Order` explicitly, but the default ordering is correct for this use case.
```

## Conventions

- One feature per directory: `.features/<feature-slug>/`
- The PRD is `.features/<feature-slug>/PRD.md`
- Implementation issues are `.features/<feature-slug>/issues/<NN>-<slug>.md`, numbered from `01`

## Issue properties

Each triaged issue has three properties: a **state**, a **category**, and a **type**. They're stored as YAML frontmatter at the top of the file — a `---`-delimited block with one `key: value` line per property:

```markdown
---
status: ready-for-agent
category: enhancement
type: AFK
---

# Title of the issue

## What to build
...
```

- **state** — stored in the `status` field. Position in the triage flow. One of `needs-triage`, `needs-info`, `ready-for-agent`, `ready-for-human`, `in-review`, `wontfix`, `done`.
- **category** — stored in the `category` field. `bug` or `enhancement`.
- **type** — stored in the `type` field. Execution mode. `AFK` (agent runs to completion alone) or `HITL` (agent runs the work but pauses at human-in-loop gates: design decisions, review checkpoints, irreversible steps).

Setting a property means writing the value into the corresponding frontmatter field. Missing fields are tolerated (the viewer omits the corresponding pill); a malformed frontmatter block surfaces a parse error.

Type and state are orthogonal. Type describes *how the work is executed*; state describes *who picks it up next*. `HITL + ready-for-agent` is normal — an agent grabs the issue and gates back to the maintainer at decision points. `ready-for-human` is reserved for work an agent genuinely cannot start (external access, manual production steps).

Type is decided at triage time. `to-issues` may write a provisional value when slicing a plan; triage confirms or flips it.

## Issue references

The canonical form is `<feature>/NN` (e.g., `microsite-path-cleanup/04`).

When the current git branch matches a feature slug, `#NN` is shorthand for `<current-branch>/NN`. Off-feature branches (`main`, branches not matching any feature) require the canonical form.

Convention: feature branches are named with the feature slug.

Convention: commits that do work toward an issue reference the issue in the commit message (e.g., `#04` or `microsite-path-cleanup/04`). Tooling — for example the `review-issue` agent — relies on this to resolve "what changed for issue NN" by scanning commits on the current branch. Tracker-mutation commits posted by `/triage` already follow this; extend it to work commits.

## Default query scope

When the current git branch matches a feature slug, queries about issues default to that feature's scope (e.g., "show issues" lists only that feature's issues). Cross-feature queries require explicit specification. Off-feature branches (e.g., `main`) default to the broadest scope.

## Setting up a feature workspace

When a feature is created (typically by `/to-prd`):

- Create a git branch named with the feature slug from the current HEAD.
- If HEAD isn't the project's main branch, warn the maintainer and confirm before branching.
- If the working tree is dirty, ask the maintainer how to proceed before branching.
- Switch to the new branch.

## Committing tracker changes

Tracker mutations — publishing an issue or PRD, changing a header field, posting a comment, writing to `.out-of-scope/`, archiving a feature — end with a git commit of the changed files, so the working tree is clean when the skill returns.

- **One commit per logical operation.** A status change plus its accompanying comment is one commit. A `/to-issues` batch publishing N new issues is one commit.
- **Mixed code-and-tracker flows** (e.g. `/implement` shipping for review) produce two commits: the code commit first, then the tracker commit for the state transition that depends on it.
- **Subject line starts with `tracker:`** so these commits can be filtered (`git log --grep '^tracker:' --invert-grep`).

## Archiving a shipped feature

When every issue in a feature dir is terminal (`status: done` or `status: wontfix`) and the feature is shipped, the entire dir moves from `.features/<feature>/` to `.features/done/<feature>/`. The PRD moves with it. Per-issue completion only flips `status`; archival happens at the feature level, on explicit maintainer request (see the `triage` skill).
