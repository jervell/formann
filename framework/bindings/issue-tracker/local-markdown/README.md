# local-markdown binding

This binding represents the issue-tracker role of the lifecycle for projects whose issues, PRDs, and triage live as markdown files under `.features/<feature>/`. It pairs the agent-facing conventions consumed by lifecycle skills (`/triage`, `/implement`, `/to-issues`, …) with a single machine-callable surface (`tracker-snapshot`) for tools that need to reason about issue state programmatically — most notably the AFK runner.

## Directory contents

- **[`BINDING.md`](BINDING.md)** — the role doc. Source of truth for the agent-facing conventions: **Tracker operations** (the seven binding-agnostic verbs and their local-markdown realizations), issue conventions, frontmatter properties, references, default query scope, workspace setup, publish/fetch semantics, committing tracker changes, archiving. Adopting projects symlink `docs/formann/issue-tracker/` at this binding folder so existing references in `CLAUDE.md` and skills keep resolving via the role surface.
- **[`tracker-snapshot`](tracker-snapshot)** — executable shell script. The binding's machine-readable interface. Emits a JSON snapshot of a feature's issues with computed eligibility flags. Contract documented below.

## Authorship

The tracker has no native attribution. In this binding, edits are filesystem writes with no author metadata; in a future GitHub Issues binding, agents would post under your credentials. In both cases, the default reading is *"the maintainer wrote this"* unless the content explicitly says otherwise.

The agent-facing rule (see [`BINDING.md`](BINDING.md#authorship)) is the inversion marker: AI-produced content carries `> *This was generated by AI during <skill>.*` on its first line; maintainer-authored content carries no marker. The marker preserves the authorship boundary on every artifact in the tracker, so anyone reading later — you, or another agent — can tell at a glance which content carries maintainer authority and which is AI-produced and therefore subject to verification.

Hand-edits transfer ownership: when you hand-edit an AI-produced body-shaped artifact (issue body, PRD, agent brief, triage notes), remove the disclaimer in the same edit. You've adopted the content as your own. Comments are timeline-historical and kept verbatim — corrections go in a new comment, not an in-place edit.

## `tracker-snapshot` machine-interface contract

Two invocation forms. Both are additive: today's bare invocation (no args)
continues to exit 2 with a usage line, unchanged.

### `tracker-snapshot <feature-slug>`

**Invocation:** `tracker-snapshot <feature-slug>` — e.g., `bash framework/bindings/issue-tracker/local-markdown/tracker-snapshot afk-runner`. The runner reaches the script via the role surface (`$HOST_REPO/docs/formann/issue-tracker/tracker-snapshot`).

**Output:** a JSON document on stdout. Exit code is `0` on success, including when individual issues are malformed.

**Schema:**

```json
{
  "feature": "<slug>",
  "issues": [
    {
      "ref": "<feature>/NN",
      "status": "<frontmatter status>",
      "category": "<frontmatter category>",
      "type": "<frontmatter type>",
      "blocked_by": ["<feature>/NN", "<other-feature>/NN", ...],
      "eligible": <bool>
    }
  ]
}
```

- `feature` — the slug as passed on the command line.
- `issues` — entries in filename order (lexicographic on `NN-<slug>.md`).
- `ref` — canonical `<feature>/NN`, derived from the directory and the issue's `NN` filename prefix.
- `status`, `category`, `type` — the verbatim frontmatter values. Missing keys produce empty strings.
- `blocked_by` — the canonical issue refs found in the `## Blocked by` section, in source order. References are matched as `<feature-slug>/<NN>` (with optional surrounding backticks). Bare `#NN` and other shorthands are not expanded — issue authors are expected to write canonical refs.
- `eligible` — boolean computed by the binding.

**Eligibility rule.** `eligible` is `true` iff *all three* hold:

1. `status == "ready-for-agent"`, and
2. `type == "AFK"`, and
3. every entry in `blocked_by` resolves to an *in-feature* issue with `status == "done"`.

Any other combination — including `ready-for-agent + HITL`, any non-`ready-for-agent` status, or unmet/unresolved blockers — yields `eligible: false`.

**Cross-feature blockers.** A blocker whose feature segment differs from the queried feature appears verbatim in `blocked_by` and forces `eligible: false`. The script does not read other features' issue directories, so it cannot verify their `status`. When a binding eventually needs that, the contract grows; for now the conservative outcome is "not eligible".

**Malformed frontmatter.** When an issue's frontmatter block is malformed (missing closing `---`, non-`key: value` lines, etc.), the issue still appears in `issues[]` with a `parse_error: "<short diagnostic>"` field added and `eligible: false`. The script does not exit non-zero just because one issue is malformed — the runner needs to see the snapshot to skip past bad files.

**What the script does *not* do.** It does not modify any files. It does not follow cross-feature blocker references. It does not interpret natural-language phrases like "None — can start immediately" — those simply contain no canonical refs and yield an empty `blocked_by`.

### `tracker-snapshot --list`

**Invocation:** `tracker-snapshot --list` — no feature-slug argument.

**Output:** a JSON array of active feature slugs on stdout. Exit code is `0` on success (including when no active features exist).

**Schema:**

```json
["slug-one", "slug-two", "slug-three"]
```

- Items are direct subdirectory names of `.features/`, excluding `done/`.
- Ordering is deterministic (lexicographic by directory name).
- Non-directory entries (files) directly inside `.features/` are ignored.

**Local-markdown realization:** `find .features/ -mindepth 1 -maxdepth 1 -type d | sort`, then filter out `done/`.

**What the script does *not* do.** It does not verify that each returned slug has a valid issue directory structure, that any issues exist, or that the feature's branch exists on the host. It returns slugs for directories that exist — further per-feature gating is the runner's responsibility.

**Consumer note.** The runner persists this output to `<run-dir>/discovery.json` for forensics. The runner validates it parses as a JSON array; non-array output causes `preflight-abort: discovery`.

## Sandbox compatibility

Any binding implementation must ensure its tracker-write operations are reachable from inside the runner's sandbox — the AFK runner dispatches `/implement` inside a Docker container with restricted filesystem and network access. A binding that needs the host filesystem, the host's `~/.gitconfig`, the LAN, or credentials that aren't passed in explicitly will silently fail when invoked from the runner.

Concretely, every binding declares what its writes need:

- **Local-markdown** — satisfied trivially. Writes are file edits and `git commit` invocations against the runner-checkout, which the sandbox mounts at `/repo`. No network, no host credentials.
- **A future GitHub Issues binding** — writes need network egress to `api.github.com` and a token. The binding doc must state both, and the runner's sandbox network policy and credential-passing must accommodate them. Both layers — the binding doc *and* the runner config — share responsibility for keeping writes reachable.

When in doubt, the test is operational: invoke the binding's write paths from inside the runner's container against a scratch feature and confirm the change appears.
