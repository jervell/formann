# Glossary rename

> *This was generated by AI during to-prd.*

## Problem Statement

The canonical vocabulary file at a consumer's repo root is named `CONTEXT.md`, which leans on DDD's "bounded context" framing to signal what's inside. That signal is invisible to readers who don't already know DDD, and "context" is overloaded in LLM tooling — it collides with context window, conversation context, and context engineering. An agent reading a repo cold and needing to look up project-specific terminology does not naturally land on `CONTEXT.md`; a human contributor unfamiliar with DDD has to open the file to discover that it is, in plain terms, a glossary.

Compounding the naming cost, the framework also supports an optional multi-context layout (`CONTEXT-MAP.md` at the root pointing at per-context `CONTEXT.md` files under subdirectories). The producer-side format spec, the `grill-with-docs` skill prompt, and the consumer-side `framework/domain.md` all carry parallel single-vs-multi branches. No current consumer uses the multi-context layout; the branches exist as a complexity tax that future readers must traverse before they reach the single path that anyone actually uses.

## Solution

Rename the artifact at the consumer root from `CONTEXT.md` to `GLOSSARY.md`. The filename now states directly what is inside — a glossary of the project's canonical vocabulary — without requiring DDD literacy from the reader. Inside the file, the inner H2 where terms live becomes `## Terms` (replacing `## Language`), and `Glossary` is added as a Formann domain term, with an `_Avoid_:` line redirecting future readers and agents from "context", "CONTEXT.md", "context map", and "ubiquitous language doc".

Drop multi-context as a supported model. There is exactly one shape from this change forward: a single `GLOSSARY.md` at the consumer's repo root. Projects with multiple internal sub-domains group their terms under H3 subheadings inside that single file (the existing "group terms under subheadings when natural clusters emerge" rule covers this). The format spec, the `grill-with-docs` skill prompt, and `framework/domain.md` collapse their parallel single/multi-context branches into one path. Capture the rename and the deliberate scope reduction as ADR 0005, so the next reader who wonders "why GLOSSARY?" or "what about bounded contexts?" finds the answer without reopening the design.

## Implementation Decisions

- The artifact at a consumer's repo root is renamed: `CONTEXT.md` → `GLOSSARY.md`. The rename is performed with `git mv` so history follows the file. There is no other supported filename or location.
- Inside `GLOSSARY.md`, the H2 heading currently labelled `## Language` becomes `## Terms`. Term entries continue to use the existing `**Name**:` + body + `_Avoid_:` shape defined in the format spec.
- `Glossary` is added as a Formann domain term inside `GLOSSARY.md`. Its `_Avoid_:` line lists `context`, `CONTEXT.md`, `context map`, and `ubiquitous language doc`. The term joins the existing **Domain docs** cluster alongside concepts like **Skill**, **Agent**, **Binding**.
- The producer-side format spec is renamed: `framework/skills/grill-with-docs/CONTEXT-FORMAT.md` → `framework/skills/grill-with-docs/GLOSSARY-FORMAT.md`. The "Single vs multi-context repos" section (including its `CONTEXT-MAP.md` instructions and skill-inference logic) is removed entirely.
- The `grill-with-docs` skill prompt (`framework/skills/grill-with-docs/SKILL.md`) drops its multi-context block. Its remaining file-structure mention becomes a single one-line prose description.
- `framework/domain.md` drops the "This is a single-context repo:" framing and its accompanying ASCII tree. The opening list points at `GLOSSARY.md` instead of `CONTEXT.md`. File structure is described in one line, without a diagram.
- Every live reference to `CONTEXT.md` in framework documentation is rewritten to `GLOSSARY.md`. The affected files are: `framework/lifecycle.md`, `framework/inbox.md`, `framework/agents/review-issue.md`, `framework/agents/review-feature.md`, `README.md`, `CLAUDE.md`, and `installer/templates/claude-md-snippet.md`.
- A new ADR is added at `docs/adr/0005-single-glossary-per-consumer.md` in the existing minimal ADR shape (title + 1–3 sentence paragraph), recording both the rename and the deliberate scope reduction.
- `.features/.archived/` snapshots are not touched. Two archived files still mention `CONTEXT.md`: `.features/.archived/self-install/PRD.md` and `.features/.archived/self-install/issues/02-readme-with-dependency-information.md`. They are preserved as point-in-time historical records.
- The installer ships the new vocabulary to fresh consumers via the rewritten `installer/templates/claude-md-snippet.md`. No migration logic, no compat shim, no detection of legacy `CONTEXT.md` files in target repos.
- The `grill-with-docs` skill's own name is unchanged. Its identity is "grilling with the help of docs"; the docs in question are now `GLOSSARY.md` and `docs/adr/`, but the skill name does not encode the artifact name.

## Testing Decisions

- A good test for this work tests an externally-visible behaviour, not an implementation detail. For this feature, the only externally-visible behaviours are: the file at the consumer's repo root exists at the new name; framework skills resolve their documented references to that file; the installer's template ships the new name to fresh consumers.
- No automated tests are added. The project has no test infrastructure for skill prompts or framework documentation files. The existing tests (`installer/tests/install.bats`, `installer/tests/self-install.bats`, the runner's bash tests) cover script behaviour and runner orchestration; none of them asserts on the text of `installer/templates/claude-md-snippet.md`, and adding such an assertion would test a documentation detail that is expected to change rather than a stable contract.
- Verification is manual:
  - Confirm `GLOSSARY.md` exists at the repo root after `git mv`, with the renamed `## Terms` heading and a `Glossary` entry whose `_Avoid_:` line lists the legacy terms.
  - Confirm `git log --follow GLOSSARY.md` traces back to the original `CONTEXT.md` commits, so history is preserved.
  - Spot-check that `framework/lifecycle.md`, `framework/inbox.md`, the two review agents, `README.md`, `CLAUDE.md`, and the installer snippet all reference `GLOSSARY.md` and not `CONTEXT.md`. A `grep -rn "CONTEXT.md" --include='*.md'` from the repo root returns only `.features/.archived/` matches.
  - Spot-check that `grep -rn -i "multi.context\|CONTEXT-MAP\|single.context" --include='*.md'` from the repo root returns no live matches.
  - Invoke `/grill-with-docs` against a fresh fixture and confirm it creates `GLOSSARY.md` rather than `CONTEXT.md` when the first term is resolved.
- Prior art for what a testable surface looks like in this repo: `installer/tests/install.bats` exercises `installer/install.sh` end-to-end against fixture consumer repos, asserting symlink shape and gitignore content. That shape applies to script behaviour; this rename is doc-shaped and does not fit it.

## Out of Scope

- Backwards-compatibility or one-shot migration logic in the installer to handle target repos that already have a `CONTEXT.md`. The cutover is clean; consumers rename manually with `git mv`.
- Reviewing, rewriting, or updating `.features/.archived/` historical snapshots. They are point-in-time records.
- Linting or tooling that enforces the presence of `GLOSSARY.md` in consumer repos. The producer skill (`/grill-with-docs`) continues to create it lazily; no enforcement is added.
- Renaming any code-side artifacts (the runner, the installer script, binding code, or the `grill-with-docs` skill itself). The rename is doc/prompt-shaped only.
- Reviving multi-context support in any form, including via subdirectory `GLOSSARY.md` files. Sub-domains group inside a single glossary under H3 subheadings; the file-system shape is fixed at one glossary per consumer.
- Cross-linking inside `GLOSSARY.md` (e.g. anchor links from term definitions to related terms). The format stays as it is today.

## Further Notes

- The decision and reasoning are captured in ADR 0005 (`docs/adr/0005-single-glossary-per-consumer.md`). The ADR is the durable artifact a future reader will find when they wonder why this rename happened.
- `Glossary` is added as a first-class Formann domain term inside the renamed file. This is recursive — the glossary defines what a glossary is — but it is also the most discoverable place for the `_Avoid_:` redirect to live. An agent searching the glossary for any of the legacy terms lands on the canonical name.
- The `_Avoid_:` line on the `Glossary` term entry is the primary durable mechanism for redirecting future agents and humans away from legacy terminology. It is more robust than a deprecation comment on the file itself: it lives inside the file's defined structure and is consumed by the same `Use the glossary's vocabulary` rule in `framework/domain.md` that already binds every skill.
- Existing PRD and skill text refers to `CONTEXT.md` as "the domain glossary" in parentheticals (e.g. ``CONTEXT.md` (domain glossary)`). With the rename, those parentheticals become tautological and are removed during the sweep — the filename is the description.
