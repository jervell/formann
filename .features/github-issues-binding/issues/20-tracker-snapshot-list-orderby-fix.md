# Fix `tracker-snapshot --list` GraphQL `orderBy` — invalid `NUMBER` enum value bricks the runner

> *This was generated by AI during to-issues.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

The `--list` GraphQL query in `framework/bindings/issue-tracker/github-issues/tracker-snapshot:103-114` orders results by `orderBy: {field: NUMBER, direction: ASC}`. `NUMBER` is not a valid `IssueOrderField` enum value — the only valid values are `CREATED_AT`, `UPDATED_AT`, and `COMMENTS` (verified via GitHub GraphQL introspection on the live API). Live execution rejects the query with `argumentLiteralsIncompatible`; the script exits with `_EXIT_TRANSIENT` after 3 retries. This bricks the AFK runner's feature discovery the moment the binding ships, because `tracker-snapshot --list` is the runner's first call against live GitHub.

The unit tests don't catch this: the `gh` shim at `tests/tracker-snapshot.bats:21-50` serves canned JSON regardless of query content — it never validates GraphQL syntax or enum values. `BINDING.md:101` documents the same broken query, so the contract itself is wrong. The test fixture at `tests/fixtures/list-archived-excluded/recorded-responses/graphql-response.json` happens to return issues already in `#N` order, so the test at `tests/tracker-snapshot.bats:154-163` cannot distinguish an API-side sort from an in-script sort — it's the structural reason the bug slipped past CI.

PRD US-15 (line 38) is the authoritative contract: `tracker-snapshot --list` returns features ordered by issue number ascending. The fix is to drop `orderBy` from the query and sort in jq via `sort_by(.number)` before extracting pairs. The pipeline at `tracker-snapshot:122-133` already projects `.number`, so the sort is trivially cheap (100-element max). `CREATED_AT` is not a safe substitute: transferred issues preserve original `createdAt` while being renumbered, so `CREATED_AT` and `#N` can diverge — only an explicit `.number` sort honors the PRD contract under all conditions.
