# Archive verb (`/triage` archive against the GH binding)

> *This was generated by AI during to-issues.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

Document and realize the **Archive** verb for the GH binding. `/triage`'s archive action under this binding closes the parent issue with `state_reason=completed` and applies `formann:archived`.

Mechanics:

- `gh issue close <parent-N> --reason completed`
- `gh issue edit <parent-N> --add-label formann:archived`

Effects:

- `tracker-snapshot --list` filters on `is:open label:formann:feature`, so closed parents drop out naturally — archived features don't appear in feature discovery.
- The `formann:archived` label is the durable archive marker visible after reopen (closed-state could be reversed; the label survives reopens as the intentional archive signal).
- Sub-issues remain in whatever state they had at archive time (typically all `done` / `wontfix`).

`/triage`'s SKILL.md stays binding-agnostic; the archive realization lives in BINDING.md.

### Bug-hunt — One substantive issue found

**Location:** `/repo/framework/bindings/issue-tracker/github-issues/BINDING.md`, lines 372–373

**Problem:** The pre-flight parent lookup in the Archive section calls the **Read the feature** verb directly — `gh issue list --label "formann:feature" --label "formann:slug:<slug>" --state open ...` — and stops with an error if no open parent is found. However, the non-terminal guard immediately after (lines 374–378) calls `tracker-snapshot <slug>`, which also queries `states: [OPEN]` for the parent issue. Since `tracker-snapshot <slug>` returns an empty `{"feature":"<slug>","issues":[]}` when the parent is not open (per the Snapshot contract, line 497–499), the non-terminal guard would see no sub-issues and no terminal-state violations — and would therefore **not** fire the refusal.

This means the non-terminal guard is a no-op if the parent was already closed before the archive attempt. The guard is only effective when the parent is still open. In that context it works correctly. However, the documentation doesn't acknowledge this ordering dependency between the two pre-flight checks. The first pre-flight guard stopping on "no open parent" protects the second guard from being vacuously bypassed, so they are sequentially dependent. The overall flow is safe (the first check catches the closed-already case and stops before the second check runs), but the second guard's protection only holds because the first guard runs first. This ordering dependency is implicit and undocumented.

This is a minor documentation completeness issue, not a behavioral bug — the sequence as written is safe.
