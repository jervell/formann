# Fix same-day same-kind comment suffix algorithm in BINDING.md

> *This was generated by AI during to-issues.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

The recipe at `framework/bindings/issue-tracker/github-issues/BINDING.md:234-241` documents how to determine the `(N)` suffix on same-day same-kind comments:

```
gh issue view <N> --json comments \
  --jq '[.comments[].body | split("\n")[0]] | map(select(startswith("### <Kind> — <YYYY-MM-DD>"))) | length'
```

This `startswith` check undercounts past the first collision: once `### Implementation (2) — 2026-05-16` exists, it no longer starts with the un-suffixed prefix `### Implementation — 2026-05-16` (character 17 is `(` vs `—`), so the third call sees `count == 1` and posts another `(2)` — producing a duplicate suffix. The bug recurs every third same-day same-kind comment.

Replace `startswith` with a pattern/regex match that counts both un-suffixed and `(N)`-suffixed prior headings. For example:

```
[.comments[].body | split("\n")[0]] |
  map(select(test("^### <Kind>( \\([0-9]+\\))? — <YYYY-MM-DD>$"))) | length
```

Update the BINDING.md recipe accordingly and add an example walking through three same-day Implementation comments to show the suffix progression `unsuffixed → (2) → (3)`.
