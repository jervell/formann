# Implement `#N` ascending sub-issue tiebreaker (and remove the misleading comment)

> *This was generated by AI during to-issues.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

PRD user-story 12 and `BINDING.md §Ordering rules` both promise: *"`#N` ascending is applied as a stable tiebreaker within equal-priority groups."* The current `tracker-snapshot` (github-issues binding) never sorts — sub-issues are emitted in whatever order the GraphQL `subIssues` connection returns them. The inline comment at `tracker-snapshot:233-236` describes a `sort -n` step that doesn't exist in the code:

```bash
# The subIssues connection is already in priority order; we preserve that
# order and use #N ascending as a stable tiebreaker (applied via sort -n
# on a secondary key, but since each position is unique it never changes
# the primary ordering).
```

Triage will pick the realization: either

- **(a)** Implement a defensive stable sort that uses `#N` ASC as a tiebreaker on any equal-priority entries the GraphQL response happens to return in non-deterministic order. The connection's existing order is preserved otherwise.
- **(b)** Document explicitly that the `subIssues` connection's order is treated as already-tiebroken (each UI position is unique), `#N` ASC is unreachable in practice, and remove the false claim from BINDING.md and the comment.

Either way: code, comment, BINDING.md, ADR-0005 (if applicable), and at least one fixture must agree on what the binding actually does. The existing `multi-issue-priority-tiebreaker` fixture returns `[#10, #7, #15]` in API order and the golden preserves that exact order — it does not exercise any tiebreaker.
