# Fix SUMMARY.md ref handling for binding-native refs

> *This was generated by AI during to-issues.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

Three SUMMARY-formatting sites in `framework/runner/run-the-queue.sh` regex-split refs on `/`, which breaks under the github-issues binding (refs of shape `#42`):

- **`format_summary_md`** (≈lines 358-369): `slash = index(ref, "/")` — for `#42`, `slash == 0`, so `nn = ref = "#42"` and the link becomes `[#42.log](#42.log)` while the actual file on disk is `42.log`. Every single-feature SUMMARY emits broken links under GH.
- **`print_multi_feature_records`** (≈lines 617-634): filters dispatch records via `[[ "$d_ref" == "$feature"/* ]]`. GH refs never contain `/`, so every drained GH-binding feature emits a table with zero `I` rows.
- **`format_multi_feature_summary_md`** (≈line 434): uses `ref` verbatim as the log path; under GH this is doubly wrong (file lives at `<feature>/42.log`; the link points at `#42.log`).

Refactor the formatters to consume binding-agnostic `nn` and `feature` fields the dispatch path already has in scope. The simplest path is to widen the `RUN_DISPATCHES` record schema from `ref|outcome|duration|review_present` to `feature|nn|ref|outcome|duration|review_present` so the formatters never have to recover `nn` from the ref string. Plumb `feature` and `nn` through `record_dispatch` and the existing call sites in `dispatch_one`.

## Acceptance criteria

- [ ] Single-feature SUMMARY.md emits a working `[<nn>.log](<nn>.log)` link under both bindings (verified with a fixture exercising a GH-shaped ref).
- [ ] Multi-feature drain SUMMARY.md emits the expected `I` rows per drained feature regardless of ref shape (GH-binding features no longer render empty tables).
- [ ] All `record_dispatch` call sites in `dispatch_one` are updated together — no half-refactored call.
- [ ] Existing local-markdown SUMMARY rendering is byte-identical for `<feature>/NN`-shaped refs (regression coverage).
- [ ] Bats coverage in `framework/runner/tests/run-the-queue.bats` exercises both `format_summary_md` and `format_multi_feature_summary_md` with `#N`-shaped refs.

**Summary:** Three SUMMARY-formatting sites in `run-the-queue.sh` split refs on `/`, silently breaking under the github-issues binding (refs of shape `#N` carry no `/`). Widen the dispatch record schema so the formatters consume binding-agnostic `nn` and `feature` fields directly, instead of trying to recover them from the ref string.

**Current behavior:**
The runner stores per-issue outcomes in the `RUN_DISPATCHES` array as `ref|outcome|duration|review_present`. SUMMARY formatters recover `nn` from the ref by indexing on `/`:

- `format_summary_md` derives `nn = substr(ref, slash+1)` — for a GH ref `#42`, `slash == 0` and `nn` falls back to the whole ref `#42`, producing the broken link `[#42.log](#42.log)` while the actual log file is `42.log`.
- `print_multi_feature_records` filters dispatch rows using `[[ "$d_ref" == "$feature"/* ]]`. GH refs never contain `/`, so the predicate never matches; every drained GH-binding feature emits a table with zero `I` rows.
- `format_multi_feature_summary_md` uses raw `ref` as the log basename — wrong under GH on both axes (file lives at `<feature>/42.log`; the link points at `#42.log`).

Local-markdown's `<feature>/NN` shape happens to satisfy both predicates, masking the breakage entirely under the legacy binding.

**Desired behavior:**
The dispatch record schema carries `feature` and `nn` alongside `ref` so formatters never reparse the ref string. Single-feature and multi-feature SUMMARY rendering produces correct links and populated tables under both bindings. Output under local-markdown's `<feature>/NN` refs is byte-identical to today's.

**Key interfaces:**

- `RUN_DISPATCHES` record schema — widen so every formatter has `feature` and `nn` available without parsing `ref`. All `record_dispatch` call sites in `dispatch_one` updated together; no half-refactored call site.
- `format_summary_md` — composes link basenames from `nn`.
- `print_multi_feature_records` — filters by `feature` field equality, not ref-prefix matching.
- `format_multi_feature_summary_md` — composes per-feature log paths from `feature` and `nn`.

**Acceptance criteria:**

- [ ] Single-feature SUMMARY emits `[<nn>.log](<nn>.log)` correctly under both bindings.
- [ ] Multi-feature drain SUMMARY emits the expected `I` rows per drained feature regardless of ref shape.
- [ ] All `record_dispatch` call sites in `dispatch_one` carry the widened fields.
- [ ] Local-markdown SUMMARY rendering is byte-identical for `<feature>/NN`-shaped refs (no regression).
- [ ] Bats coverage in `framework/runner/tests/run-the-queue.bats` exercises both `format_summary_md` and `format_multi_feature_summary_md` with `#N`-shaped refs.

**Out of scope:**

- Changing binding-native ref shapes (`#N` for GH, `<feature>/NN` for local-markdown).
- Reformatting SUMMARY.md output structure beyond what's necessary to make links and tables correct.
