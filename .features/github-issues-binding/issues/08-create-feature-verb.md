# Create-a-feature verb (`/to-prd` against the GH binding)

> *This was generated by AI during to-issues.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

Document and realize the **Create a feature** verb for the GH binding. This is the entry point for `/to-prd` and the first verb that writes to GitHub.

Verb realization (one logical operation, four steps):

1. **Pre-flight slug uniqueness check.** Run `gh issue list --label formann:slug:<slug> --state all`. If any match exists, refuse with a clear error naming the existing issue. The maintainer picks a different slug.
2. **On-demand slug-label creation.** `gh label create formann:slug:<slug> --force` — idempotent. The static label namespace is assumed pre-created via `bootstrap-labels` (see slice 02); the per-feature slug label is created on the fly.
3. **Parent issue creation.** `gh issue create --title <title> --body @prd.md --label formann:feature,formann:slug:<slug>`. Title goes in the native title field; PRD body goes in the issue body (no `# Title` duplication).
4. **Feature branch creation.** Create and switch to the feature branch (slug → branch name), per the existing local-markdown convention. This step is binding-agnostic and unchanged.

Slug-length validation: the binding's 50-char label cap leaves 37 chars for the slug (`formann:slug:` is 13 chars). `/to-prd` validates slug length at pick time and rejects early.

`/to-prd`'s SKILL.md already says "Set up the feature workspace per `docs/formann/issue-tracker/BINDING.md`" and "publish per `docs/formann/issue-tracker/BINDING.md`" — so this slice is primarily BINDING.md content. The skill stays binding-agnostic.

### Finding 1 — Pre-flight uniqueness check uses a different label filter than tracker-snapshot

**Location:** `/repo/framework/bindings/issue-tracker/github-issues/BINDING.md`, line 86

**Problem:** The Step 1 command is:
```
gh issue list --label "formann:slug:<slug>" --state all --json number,title
```

The Agent Brief (issue line 70) explicitly specifies that this pre-flight check and tracker-snapshot's slug-uniqueness check should share "the same query shape (`gh issue list --label formann:slug:<slug>` AND `formann:feature`)." The tracker-snapshot script (line 192) filters by BOTH labels — `labels: ["formann:feature", $slugLabel]`. The documented pre-flight check uses only the slug label.

The practical consequence is that the pre-flight check is overly broad: any issue (sub-issue, hand-created issue, etc.) carrying `formann:slug:<slug>` — even one that is not a feature parent — would cause the pre-flight to refuse. This is a false-positive risk rather than a missed-collision risk. It is still a divergence from the specified behavior, and the Agent Brief's explicit "same query shape" contract is not met.
