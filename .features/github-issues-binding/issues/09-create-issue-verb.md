# Create-an-issue verb (`/to-issues` against the GH binding)

> *This was generated by AI during to-issues.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

Document and realize the **Create an issue (slice)** verb for the GH binding. This is the entry point for `/to-issues` and lets the binding capture the feature → sub-issue hierarchy natively at creation time.

Verb realization (two steps, one logical operation):

1. **Create the sub-issue.** `gh issue create --title <title> --body @issue.md --label formann:status:needs-triage,formann:category:<cat>,formann:type:<type>`. The body follows the issue template with one binding-specific tweak: under the GH binding, sub-issue bodies **skip the `## Parent` section**. GitHub's native sub-issue panel surfaces the parent; the Read-the-feature verb (slice 10) gives skills programmatic access.
2. **Link to the parent.** `gh api graphql -f query='mutation { addSubIssue ... }'` attaches the newly-created issue as a sub-issue of the parent. The hierarchy is captured natively at creation time so the GitHub UI shows it immediately.

`/to-issues`'s SKILL.md already says "write and publish a new issue per `docs/formann/issue-tracker/BINDING.md`" — this slice updates BINDING.md to describe the per-binding `## Parent`-omission and the `addSubIssue` step. The skill stays binding-agnostic.
