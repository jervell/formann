# Runner refactor: `(feature, N)` plumbing and `binding_native_ref` helper

> *This was generated by AI during to-issues.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

Refactor the runner so it never regex-parses `<feature>/NN` to derive the feature and never assumes the binding-native ref shape. This is binding-agnostic prep work: it lands before any GH-specific code and the local-markdown path stays identical end-to-end.

Three call sites consume the ref by regex-parsing today; each switches to consuming a `(feature, N)` pair plus, where needed, the binding-native ref form retrieved via a new helper:

1. **`dispatch_one`** — the snapshot's `feature` field and the issue's `nn` are both in scope. Pass them explicitly to downstream helpers instead of regex-parsing the ref. When the binding-native ref needs to be passed into a binding script, resolve it via `binding_native_ref`.
2. **`write_abort_flag`** — `.runner-state/aborted/<feature>/<NN>`. Continues to use `<feature>/<N>` as the filesystem path; both halves come from the `(feature, N)` pair, never from regex-parsing the binding-native ref.
3. **Classifier invocations** — wherever the runner passes the ref into the classifier, it now passes `(feature, N)` and resolves the binding-native ref via the helper if the classifier needs it.

New helper: `binding_native_ref <feature> <N>` looks up the binding-native ref by indexing into the current snapshot's issues array. The runner's CLI argument parser (`--issue <feature>/<N>`, regex `^([a-z0-9][a-z0-9-]*)/([0-9]+)$`) is unchanged — that's the **portable** form for operator ergonomics.

Also in this slice (binding-agnostic, related):

- Move HEAD-archive logic out of the `take_snapshot` wrapper and into `framework/bindings/issue-tracker/local-markdown/tracker-snapshot`. The wrapper shrinks to a thin shell-out; each binding owns its "current state of the world" definition.
