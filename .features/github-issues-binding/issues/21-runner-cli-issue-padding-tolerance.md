# Make runner `--issue` lookup tolerant of `nn` zero-padding differences

> *This was generated by AI during to-issues.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

The runner's `--issue <feature>/<NN>` argparse at `framework/runner/run-the-queue.sh:728-735` extracts the trailing integer verbatim into `$ISSUE_NN`, with no normalization. The two issue-tracker bindings emit `nn` with different padding conventions:

- **local-markdown** emits `nn` lifted verbatim from the filename regex `^([0-9]+)-...` — e.g., `"03"` for `03-foo.md`. Zero-padded.
- **github-issues** emits `nn` from the raw GraphQL `.number` field — e.g., `"3"` for issue `#3`. Unpadded.

`run_single` at `run-the-queue.sh:1709-1710` does exact-string jq compare:

```bash
entry="$(jq -c --arg n "$ISSUE_NN" \
  'first(.issues[] | select(.nn == $n)) // empty' <<<"$snap")"
```

So a maintainer who types `--issue github-issues-binding/03` against the github-issues binding (snapshot has `"3"`) — or `--issue self-install/3` against local-markdown (snapshot has `"03"`) — gets refused with `single-dispatch refused: missing — ref '…/03' is not present in feature '…'`. The diagnostic at line 1713 does not hint that the issue exists but with a different padding.

This is a UX footgun, not a correctness bug: typing the right padding works. But cross-binding muscle memory or copying a number from the GitHub UI (`#7`) and typing it against a local-markdown feature (filename `07-foo.md`, snapshot `"07"`) hits it. The new binding doubles the surface and the asymmetry makes the diagnostic more confusing.

Fix: compare numerically in the lookup so padding stops mattering. The contract (per `BINDING.md` field note for `nn`) is "the integer half of the runner's CLI ref form `<feature>/<NN>`" — always integer-valued — so numeric comparison is honest. Use `jq`'s `tonumber` on both sides of the compare.

## Acceptance criteria

- [ ] `framework/runner/run-the-queue.sh:1709-1710` (`run_single` lookup) compares `nn` numerically: `select((.nn | tonumber) == ($n | tonumber))`.
- [ ] Argparse at `run-the-queue.sh:728-735` is unchanged — the regex still accepts `^[0-9]+$` for the NN portion; no other changes needed there.
- [ ] `framework/runner/tests/run-the-queue.bats`: regression tests cover both directions — `--issue feature/03` resolves a snapshot entry with `nn:"3"`; `--issue feature/3` resolves a snapshot entry with `nn:"03"`.
- [ ] All existing `run-the-queue.bats` tests continue to pass.
- [ ] No other runner call sites need changing: `binding_native_ref` (line 269), `run_loop` (line 1808), and `feature_has_dispatchable_ref` (line 2034) all source `nn` from the snapshot itself, not from CLI input — they cannot have padding mismatches and should not be modified.

**Summary:** One-line jq change in `run_single`'s lookup, plus two regression tests. The lookup compares `nn` as a string today; it should compare as a number.

**Current behavior:**

- The runner's `--issue <feature>/<NN>` argparse regex (`^([a-z0-9][a-z0-9-]*)/([0-9]+)$`) matches both `feature/3` and `feature/03`. The captured `${BASH_REMATCH[2]}` is the literal user input.
- local-markdown emits `nn` zero-padded (verbatim from filename); github-issues emits `nn` unpadded (raw GraphQL `.number`).
- `run_single`'s jq lookup `select(.nn == $n)` does exact-string compare, so the strings must match exactly.
- A user who types the "wrong" padding for the active binding sees the issue refused with `single-dispatch refused: missing`, even though the issue exists.
- The error message does not surface padding as a possible cause.

**Desired behavior:**

- `--issue feature/03` resolves a snapshot entry with `nn:"3"` (and vice versa). The runner stops caring about leading zeros in the CLI input.
- Behavior is identical for callers that already type the matching padding.
- The error message is unchanged — the failure mode it describes (issue genuinely missing) is still the right failure mode when no entry matches under either padding.

**Key interfaces:**

- `framework/runner/run-the-queue.sh:1709-1710` — the only call site that receives potentially-mismatched-padding `nn` from the user. The other three `.nn`-consuming sites (`binding_native_ref`, `run_loop`, `feature_has_dispatchable_ref`) source `nn` from the snapshot itself and never see user input, so they don't need changing.
- `framework/runner/tests/run-the-queue.bats` — add two regression tests, one per direction. Use the existing fixture pattern (snapshot stub + CLI invocation + assertion on exit code and trailing reason).

**Acceptance criteria:**

- [ ] `--issue feature/03` against a snapshot with `nn:"3"` resolves the entry and proceeds with dispatch (or fails at the next eligibility check, not at the lookup).
- [ ] `--issue feature/3` against a snapshot with `nn:"03"` resolves the entry and proceeds with dispatch.
- [ ] The runner does not regress on the existing happy paths (matching padding under either binding).
- [ ] All existing `run-the-queue.bats` tests pass.

**Out of scope:**

- Argparse normalization (storing a normalized form alongside the raw). Numeric compare in jq is a cleaner fix and avoids carrying two variables through the runner.
- Improving the "missing" diagnostic to hint at padding. With numeric compare, padding is no longer a failure mode worth diagnosing.
- Normalizing what the bindings emit. Each binding's `nn` value reflects its native identity (filename sequence vs. GitHub issue number); changing emit shape risks downstream surprises.

## Findings

**Finding 1: The fix is incomplete — `binding_native_ref` still uses string comparison when called from the `run_single` path**

Location: `/repo/framework/runner/run-the-queue.sh`, lines 1445-1451 (in `dispatch_one`) and line 269 (in `binding_native_ref`)

Problem: `run_single` calls `dispatch_one "$ISSUE_FEATURE" "$ISSUE_NN"` at line 1745, passing the raw user-typed `ISSUE_NN`. Inside `dispatch_one`, line 1446 calls `binding_native_ref "$feature" "$nn" "$pre_json"` with that same user-typed value. `binding_native_ref` does `select(.nn == $n)` — string comparison — against the pre-dispatch snapshot. If the user typed `--issue github-issues-binding/03` but the snapshot has `nn:"3"`, this string comparison finds nothing, `binding_native_ref` returns 1, and `dispatch_one` sets `RUN_STOP_REASON="snapshot-failed-mid-dispatch:pre"` and returns 1. The `run_single` eligibility gate passes (the fix works there), but dispatch fails with a misleading "snapshot-failed-mid-dispatch:pre" instead of succeeding. The cross-padding scenario that the fix was intended to address still fails end-to-end at runtime, just at a different point.

Severity: 🔴 Critical

This is the core correctness failure: the stated goal of the issue (letting users type either padding and have dispatch succeed) is not achieved. The fix moves the failure site from the eligibility gate to the dispatch phase, and changes the error message, but does not make the feature work.

**Finding 2: Regression tests do not expose the remaining failure path**

Location: `/repo/framework/runner/tests/run-the-queue.bats`, lines 1226-1262 (both new tests)

Problem: Both new regression tests stub out `dispatch_one` (via `setup_run_single_test` at line 1126-1129). They verify that the eligibility gate passes (exit 0, `RUN_STOP_REASON = "single-dispatch (success)"`, `TEST_DISPATCH_CALLED` file non-empty). But the stubbed `dispatch_one` does not call `binding_native_ref`, so the failing string comparison never executes in the test. The tests pass and claim the fix works, but the real `dispatch_one` → `binding_native_ref` path is never tested. This is not a problem with the test structure per se (the surrounding tests use the same stub pattern appropriately), but it means neither test actually validates the end-to-end padded-dispatch scenario.

Severity: 🟡 Important

The tests verify the correct layer (the eligibility gate), but they create a false confidence that the issue is fully resolved when it is not.
