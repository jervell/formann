## What to build

Stand up the `framework/bindings/issue-tracker/github-issues/` folder per ADR-0004, with the skeleton documents and the `bootstrap-labels` install-time script. This is the binding's foundation; subsequent slices fill in verb realizations on top of it.

Folder contents this slice creates:

- `BINDING.md` — skeleton with the binding's top-level shape (Authorship section, "Tracker operations" preamble, empty verb sections that later slices fill in). The cross-cutting sections (snapshot contract, body-edit residual race, sandbox compatibility, hard-limits table, GHE prerequisite) are stubs; later slices populate them as their content lands.
- `README.md` — human-facing, mirroring local-markdown's README shape (directory contents, authorship explanation, snapshot contract pointer, sandbox compatibility pointer, label-namespace bootstrap procedure).
- `bootstrap-labels` — idempotent install-time script. Creates the binding's **static** label namespace via `gh label create --force`. Dynamic per-feature labels (`formann:slug:<slug>`) are not created here.

Static label namespace this slice creates:

- `formann:feature`, `formann:archived`
- `formann:status:needs-triage`, `formann:status:needs-info`, `formann:status:ready-for-agent`, `formann:status:ready-for-human`, `formann:status:in-review`, `formann:status:wontfix`
- `formann:category:bug`, `formann:category:enhancement`
- `formann:type:afk`, `formann:type:hitl`

No `formann:status:done` label — terminal `done` uses GitHub's closed-with-`state_reason=completed` mechanism.
