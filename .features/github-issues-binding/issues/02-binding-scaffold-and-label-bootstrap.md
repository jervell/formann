**Summary:** Stand up the `framework/bindings/issue-tracker/github-issues/` folder per ADR-0004 with the BINDING.md and README.md skeletons and the `bootstrap-labels` install-time script that creates the binding's static label namespace.

**Current behavior:**
Only one issue-tracker binding exists today: `framework/bindings/issue-tracker/local-markdown/` (containing `BINDING.md`, `README.md`, and `tracker-snapshot`). The folder for the second binding (`github-issues/`) does not exist. The static label namespace required to write to GitHub Issues (`formann:feature`, `formann:archived`, all `formann:status:*`, `formann:category:*`, `formann:type:*`) does not exist on any GitHub repo — `gh issue create --label` against a non-existent label fails with HTTP 422. There is no install-time mechanism to create these labels.

**Desired behavior:**
The `framework/bindings/issue-tracker/github-issues/` folder exists per the ADR-0004 binding-folder convention. It contains three artifacts this slice creates:

1. **`BINDING.md` skeleton** — top-level shape with Authorship section, "Tracker operations" preamble, empty verb sections later slices populate. Idempotent across reruns (`--force` makes label creation a no-op when the label already exists). Exits non-zero with a clear stderr error when `gh` auth is missing, mirroring the pattern in `framework/runner/retrieve-token.sh`.

Dynamic per-feature `formann:slug:<slug>` labels are **not** created here — those land on-demand in `/to-prd`'s **Create a feature** verb (slice 08).
