## Findings

### Finding 1: "missing gh binary" test is not hermetic

**Location:** `/repo/framework/bindings/issue-tracker/github-issues/tests/bootstrap-labels.bats`, lines 113–121

**Problem:** The test removes only the fake `gh` shim from `$FAKE_BIN` but leaves `$FAKE_BIN` prepended to `PATH`. If a real `gh` binary exists anywhere else in the caller's `PATH`, `command -v gh` in `bootstrap-labels` finds it. If that real `gh` also has valid auth (via keyring or a real `GH_TOKEN` already in the environment), the script's auth guard passes and `bootstrap-labels` proceeds to call `gh label create` against a real GitHub repo — `assert_failure` then fails (the script exits 0), so the test failure would surface the problem, but the side effect (real label creation calls) occurs before the assertion. On a CI host without `gh` installed (including this repo's environment), the test passes correctly. The test is environment-dependent.

The fix is straightforward: restrict `PATH` to only `$FAKE_BIN` for the duration of this test, or use `PATH="$FAKE_BIN" run "$BOOTSTRAP_LABELS"`.

**Severity:** Minor — the test passes correctly in CI and any host without `gh` installed. The failure mode on a developer machine with `gh` + auth is a noisy test failure, not a silent false-positive.
