# `body-edit` section splicer

> *This was generated by AI during to-issues.*

## Parent

## What to build

Ship `body-edit` at `framework/bindings/issue-tracker/github-issues/body-edit`. This is the section-level body splicer that body-shaped verbs (publish agent brief, record triage notes, set blocker metadata) call.

CLI: `body-edit <issue-number> <section-heading> <new-content>`. `<new-content>` is a file path; `/dev/null` (or an empty file) signals "delete the section."

Mechanics: fetches the current body via `gh issue view`, finds or appends the target `## <section>` heading, replaces from there to the next `## ` heading (or end of body), and PATCHes the entire body back. The write is whole-body in mechanics even though it's section-level in intent — this matters for the residual race documented in BINDING.md.

Document in BINDING.md's "Body-shaped write semantics" section:

- The fetch-then-write window. A concurrent web-UI edit to **any** section of the body between fetch and write gets silently clobbered.
- The race window matters more in some flows than others: AFK runner = narrow window, interactive `/triage` = wide window. Maintainer guidance: don't hand-edit a body in the UI while a Formann skill is running on it.
- No engineered mitigation. GitHub provides no optimistic-concurrency primitive on issue bodies.

1. Fetch the current body via `gh issue view --json body`.
2. Find or append the target `## <section>` heading in memory.
3. Replace from there to the next `## ` heading (or end of body), splicing the new content in (or removing the section when new-content is empty).
4. PATCH the entire body back via `gh issue edit --body-file <tmp>` (or equivalent).

- **Section exists**: target section is replaced; surrounding sections preserved verbatim (including blank lines and trailing content).

## Findings

### Bug-hunt

**Location:** `/repo/framework/bindings/issue-tracker/github-issues/body-edit`, line 81 — `awk -v heading="$_heading_line"`

**Problem:** The `heading` variable is passed to awk via `-v`, which processes escape sequences in the value string. A section heading containing a literal backslash followed by a letter (e.g., `\n`, `\t`) would be silently mangled — `\n` becomes a newline, `\t` becomes a tab — causing the heading match (`$0 == heading`) to never fire. The agent correctly identified and avoided this issue for `content_file` (passing it as a file path, not via `-v`), but the same defect exists for the heading. In practice the named sections are "Agent Brief", "Triage Notes", "Blocked by" — none contain backslashes — so this is not triggerable in the current codebase. The asymmetry between how the two variables are handled is worth noting.

**Severity:** 🟢 Minor

---

**Location:** `/repo/framework/bindings/issue-tracker/github-issues/body-edit`, lines 80–128 (the awk script as a whole)

**Problem:** When `in_section=1` and a blank line is encountered, the line is discarded by the `else { next }` branch. This means any blank line that appears between the replaced section's content and the next `## ` heading is removed from the output, leaving no blank separator before the following heading. Example: `## Target\ncontent\n\n## Next` becomes `## Target\nnew content\n## Next`. This is a documented implementation decision ("blank lines that were part of the old target section's trailing content are not preserved"), but it can produce visually dense output in bodies that use blank lines as section separators.

More critically, none of the bats tests use fixtures with blank lines between sections, so the section-exists test passes with no `## Next` directly adjacent to `## Target` in any fixture — the "surrounding sections are preserved verbatim (including blank lines)" AC is not exercised for inter-section blank lines. The test cannot falsify a regression here.

**Severity:** 🟡 Important — not a bug per the documented decision, but the AC wording says "preserved verbatim (including blank lines)" and the test does not cover this scenario. A future caller's body with blank-line separators will lose those separators after a `body-edit` call.

---

**Location:** `/repo/framework/bindings/issue-tracker/github-issues/body-edit`, lines 73–74

**Problem:** `mktemp /tmp/body-edit-XXXXXX` hardcodes `/tmp`. On some systems the expected temp directory is `$TMPDIR` (macOS sets this to a per-user path; `/tmp` may not be writable in sandboxed or containerized environments). The conventional form is `mktemp "${TMPDIR:-/tmp}/body-edit-XXXXXX"`. This will matter when issue 06 ("sandbox env hook and script") constrains the runtime environment.

**Severity:** 🟢 Minor

---

**Location:** `/repo/framework/bindings/issue-tracker/github-issues/tests/body-edit.bats`, lines 74–85 (section-exists test) and lines 189–198 (multiple-headings test)

**Problem:** These tests check for presence/absence of strings with `[[ "$written" == *"..."* ]]` but do not verify section ordering. A body where `## Last` appeared before `## First` would pass all these assertions. The section-absent test (lines 112–114) does verify ordering via `grep -n` line-number comparison. The correctness of ordering is structurally guaranteed by awk's sequential processing, but the absent coverage means no test would catch a regression if the ordering logic were changed.

**Severity:** 🟢 Minor
