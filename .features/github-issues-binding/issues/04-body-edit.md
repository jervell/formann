# `body-edit` section splicer

> *This was generated by AI during to-issues.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

Ship `body-edit` at `framework/bindings/issue-tracker/github-issues/body-edit`. This is the section-level body splicer that body-shaped verbs (publish agent brief, record triage notes, set blocker metadata) call.

CLI: `body-edit <issue-number> <section-heading> <new-content>`. `<new-content>` is a file path; `/dev/null` (or an empty file) signals "delete the section."

Mechanics: fetches the current body via `gh issue view`, finds or appends the target `## <section>` heading, replaces from there to the next `## ` heading (or end of body), and PATCHes the entire body back. The write is whole-body in mechanics even though it's section-level in intent — this matters for the residual race documented in BINDING.md.

Document in BINDING.md's "Body-shaped write semantics" section:

- The fetch-then-write window. A concurrent web-UI edit to **any** section of the body between fetch and write gets silently clobbered.
- The race window matters more in some flows than others: AFK runner = narrow window, interactive `/triage` = wide window. Maintainer guidance: don't hand-edit a body in the UI while a Formann skill is running on it.
- No engineered mitigation. GitHub provides no optimistic-concurrency primitive on issue bodies.

**Summary:** Ship `body-edit` — the section-level body splicer that body-shaped verbs (publish agent brief, record triage notes, `## Blocked by` writes) call. Whole-body PATCH mechanics with documented residual race.

**Current behavior:**
There is no section-level body splicer for the github-issues binding. The binding folder (created in slice 02) has no body-edit script yet. GitHub's REST API edits issue bodies whole-body — there is no native section-level primitive, and no optimistic-concurrency primitive on bodies.

**Desired behavior:**
`framework/bindings/issue-tracker/github-issues/body-edit` exists, is executable, bash 3.2 compatible. CLI: `body-edit <issue-number> <section-heading> <new-content>` where `<new-content>` is a file path; `/dev/null` (or an empty file) signals "delete the section".

Mechanics:
1. Fetch the current body via `gh issue view --json body`.
2. Find or append the target `## <section>` heading in memory.
3. Replace from there to the next `## ` heading (or end of body), splicing the new content in (or removing the section when new-content is empty).
4. PATCH the entire body back via `gh issue edit --body-file <tmp>` (or equivalent).

Section semantics:

- **Section exists**: target section is replaced; surrounding sections preserved verbatim (including blank lines and trailing content).
- **Section absent**: section appended at the end of the body with the right `## <heading>` line.
- **Empty new content**: the section (heading + contents through the next `## ` or end of body) is removed; surrounding sections preserved.
- **Multiple `##` headings**: only the named section is mutated.
- **AI disclaimer preservation**: if new content starts with a `> *…*` line, it's preserved inside the section verbatim — the script doesn't strip or reorder content within a section.

BINDING.md's "Body-shaped write semantics" section documents:

- The fetch-then-write window — concurrent web-UI edits to *any* section between fetch and write get silently clobbered.
- AFK runner = narrow window (rare race); interactive `/triage` = wide window (plausible race). Maintainer guidance: don't hand-edit a body in the UI while a Formann skill is running on it.
- No engineered mitigation. GitHub provides no optimistic-concurrency primitive on issue bodies.
- The header comment in the script also names the whole-body-write mechanics so a reader of the script sees the same fact.
