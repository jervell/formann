# `body-edit` section splicer

> *This was generated by AI during to-issues.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

Ship `body-edit` at `framework/bindings/issue-tracker/github-issues/body-edit`. This is the section-level body splicer that body-shaped verbs (publish agent brief, record triage notes, set blocker metadata) call.

CLI: `body-edit <issue-number> <section-heading> <new-content>`. `<new-content>` is a file path; `/dev/null` (or an empty file) signals "delete the section."

Mechanics: fetches the current body via `gh issue view`, finds or appends the target `## <section>` heading, replaces from there to the next `## ` heading (or end of body), and PATCHes the entire body back. The write is whole-body in mechanics even though it's section-level in intent — this matters for the residual race documented in BINDING.md.

Document in BINDING.md's "Body-shaped write semantics" section:

- The fetch-then-write window. A concurrent web-UI edit to **any** section of the body between fetch and write gets silently clobbered.
- The race window matters more in some flows than others: AFK runner = narrow window, interactive `/triage` = wide window. Maintainer guidance: don't hand-edit a body in the UI while a Formann skill is running on it.
- No engineered mitigation. GitHub provides no optimistic-concurrency primitive on issue bodies.

## Acceptance criteria

- [ ] `framework/bindings/issue-tracker/github-issues/body-edit` exists and is executable.
- [ ] Section-exists case: target section is replaced; surrounding sections are preserved verbatim (including blank lines and trailing content).
- [ ] Section-absent case: the section is appended at the end of the body with the right `## <heading>` line.
- [ ] Empty-new-content case: the section (heading + contents through the next `## ` or end of body) is removed; surrounding sections are preserved.
- [ ] Multiple-`##`-headings case: only the named section is mutated; other sections at the same level stay verbatim.
- [ ] AI-disclaimer placement: if the supplied new content includes a leading `> *…*` disclaimer line, it's preserved inside the section (the script doesn't strip or reorder content within the section).
- [ ] The script calls `gh issue view --json body` to fetch, splices in-memory, and `gh issue edit --body-file <tmp>` (or equivalent) to write back. Whole-body write is documented in the script's header comments.
- [ ] Bats tests cover each acceptance criterion above. Tests use a `gh` shim to capture the fetched body and the written body without hitting the network.
- [ ] BINDING.md's "Body-shaped write semantics" section is in place and documents the residual race, the AFK-narrow vs interactive-wide windows, and the explicit guidance to maintainers.
