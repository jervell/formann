## What to build

Two pieces, one slice, because the smoke walk *is* the cutover for Formann itself:

1. **End-to-end smoke walk** on a scratch feature against `jervell/formann`'s GitHub Issues. Run the full skill chain: `/to-prd` → `/to-issues` → `/triage` → `/implement` → review → `/triage` archive. Verify every verb realization end-to-end. Record the run as a smoke artifact under `.runner-state/smoke-runs/YYYY-MM-DD-github-issues-binding.md` per `.claude/rules/runner-smoke-artifacts.md`.

2. **Formann cutover.** Flip Formann's own tracker binding from local-markdown to github-issues:
   - Run `bootstrap-labels` against the Formann repo.
   - Flip the role-surface symlink: `docs/formann/issue-tracker/` → `../../.formann/bindings/issue-tracker/github-issues` (or however the installer naturalizes it; the installer's per-Consumer binding choice is the canonical path).
   - Commit the flag-day change.
   - `.features/.archived/` stays in git history (no backfill into GitHub — per PRD's out-of-scope).
   - Future features author on GH from the start.

This slice also does the final BINDING.md sweep:

- Ensure the hard-limits table is in place (100 sub-issues per parent; 8 levels of nesting; 50-char label name; 65,536-char issue body) with failure mode and workaround for each.
- Ensure the GHE version prerequisite (sub-issues feature enabled) is stated.
- Ensure every verb shows a concrete `gh` command (US 31's full sweep).
- Ensure the list-order is binding-specific note appears.
- Ensure the web-UI comment convention noise note appears.

The slice is HITL because the smoke walk requires a maintainer driving the skill chain interactively against a real repo, and the cutover is a deliberate flag-day act.
