# Self-install

> *This was generated by AI during to-prd.*

## Problem Statement

The maintainer wants to develop further on Formann using Formann's own methodology — `/triage`, `/implement`, the AFK runner, the issue tracker. Today this isn't possible without forking the framework's contract: running `installer/install.sh .` against Formann itself produces ~17 symlinks and a `runner/Dockerfile` scaffold that, on a normal consumer, are meant to be committed (stable paths via `.formann` indirection). For Formann, committing those products pollutes `main` with redundant indirection symlinks (`.claude/skills/triage -> .formann/skills/triage -> framework/skills/triage`) that consumer-evaluators see on a fresh clone as dangling links, since `.formann` is gitignored. The alternative — leaving the installer products untracked — leaves `git status` permanently noisy. A blanket `/.claude/skills/`-style `.gitignore` rule blocks the maintainer from authoring non-framework skills locally. None of the available workarounds preserves both clean-`main` and a usable dogfood loop.

## Solution

Teach the installer to detect when it is installing Formann into itself (consumer path resolves to Formann path) and, in that mode only, manage a delimited block in `.gitignore` that lists every product the installer creates. The block is regenerated on each install run, so adding a new skill to `framework/skills/` and re-running the installer keeps the ignore list in sync. Outside-block content is preserved. A regular consumer install is unchanged — the existing `/.formann` line is the only `.gitignore` mutation, and consumers continue to commit their stable symlinks per ADR-0001. After the enhancement lands, the maintainer self-installs Formann once, commits the gitignore plus the methodology content already accumulating on `main` (`.out-of-scope/`, `.features/`), and Formann's own development from that point forward dogfoods the framework end to end.

## Implementation Decisions

- **Module structure:** six issues — (1) product enumeration extraction, (2) managed-block gitignore writer, (3) self-install detection wiring, (4) README — contributor section + dependency information, (5) self-install bootstrap (one-off, lands on `main` after the installer change), (6) Formann's own `.features/` tracker dogfood proof (this PRD plus its issues, archived once the work is done). Issues 1, 2, 3 are code changes; issue 4 is documentation; issues 5 and 6 are one-off operational steps. Issues 1→2→3 are sequential (each consumes the prior). Issue 4 is independent of the code changes. Issues 5 and 6 depend on 3 (the live installer must support self-install before the bootstrap is meaningful).

- **Self-install detection:** use bash `[ "$consumer_path" -ef "$formann_path" ]` after both paths are resolved to absolute canonical form. This handles trailing slashes, symlinks, and relative paths uniformly. The check runs once in `main()` and sets a `SELF_INSTALL` flag consumed by `update_gitignore`.

- **Managed-block markers:** the comment pair is `# === Formann self-install (managed by installer — do not edit) ===` (start) and `# === /Formann self-install ===` (end). On each install run in self-install mode: the writer finds the markers (or appends them), replaces the block between them with the freshly enumerated product list (including `/.formann`), and preserves everything outside. The instruction to "not edit" is informational — the script will overwrite manual edits inside the block, but does not validate or warn.

- **Product enumeration:** a new shell function `list_installer_products()` returns the relative paths (from consumer root) the installer would create for the current Formann checkout and role-binding selections. It walks `framework/skills/*`, `framework/agents/*.md`, `framework/rules/*`, the role-surface flat docs, and the chosen binding folders. Both the existing `link_*` functions and the new managed-block writer call this function — the existing inline traversals are replaced with calls into the enumerator. `runner/Dockerfile` is included in the product list as a real file (the gitignore needs to list it; `make_symlink` is not its creator, but it is an installer product).

- **Consumer-install behavior unchanged:** when `SELF_INSTALL` is false, `update_gitignore` behaves as today — appends `/.formann` if missing, no managed block. The existing bats test fixtures and assertions remain valid.

- **`.gitignore` for the Formann repo itself (pre-bootstrap):** currently contains `.DS_Store`, `/.idea/`, `*.iml`, `/.claude/settings.local.json`, `__pycache__/`. The bootstrap commit on `main` will add the managed block beneath these existing entries. No reordering of existing lines.

- **README.md scope:** rewritten to cover (a) what Formann is (one paragraph, links to `CONTEXT.md`), (b) consumer adoption (one-paragraph pointer to `installer/README.md`), (c) contributor setup — clone, run `installer/install.sh .`, what the dogfood loop gives you, (d) dependency information split into "to use Formann (consumer-side)" and "to develop Formann (contributor-side)" subsections. Existing README content is short; rewriting in place is cleaner than incremental edits.

- **Dependency information content:** consumer-side lists `bash`, `git`, `Maven` (per v0 assumption from ADR), and any host-level expectations the runner Dockerfile has. Contributor-side adds `bats` (for the test suite) and any tooling the framework skills themselves depend on. Versions are pinned to "what works today" — no speculative minimum-version exercise.

- **No new shell tooling:** the enumerator and the managed-block writer are pure bash, using the same constructs already present in `install.sh` (globs, arrays, parameter expansion, `printf`/`echo`, `grep`/`sed`). No `awk`, `jq`, `python`, or other runtime additions.

- **Bootstrap shape (issue 5):** runs `./installer/install.sh .` from the Formann root with `FORMANN_INSTALL_BINDING_issue_tracker=local-markdown` to bypass the interactive prompt, then `git add .gitignore && git commit` with a `tracker:`-prefixed message documenting the dogfood transition. Symlinks themselves are not committed (gitignored). `runner/Dockerfile` is not committed (gitignored). The bootstrap issue's acceptance criteria assert that `git status` is clean after install.

- **Documentation precedent:** the existing `installer/README.md` is the canonical place for installer mechanics. The top-level `README.md` (this PRD's target) is the consumer/contributor entry point and points at the installer README rather than duplicating it. CLAUDE.md retains its current contents — conventions about README being human-only stay there.

## Testing Decisions

- **What makes a good test here:** tests must FAIL when the installer produces a wrong gitignore state and PASS when the resulting `.gitignore` matches the spec. They assert on observable `.gitignore` content and on `git status` cleanliness — never on internal enumerator or writer implementation details. Same shape as the existing `install.bats` suite.

- **Modules under test:**
  - **Managed-block writer (module 2)**: highest test ROI. Pure text manipulation, easy to set up. Test fixtures: empty `.gitignore`, `.gitignore` with unrelated entries only, `.gitignore` with an existing managed block, `.gitignore` with a stale managed block (entries missing or extra), `.gitignore` with manual edits inside the block (assert overwritten on next run).
  - **Self-install end-to-end (module 3)**: fixture creates a synthetic Formann checkout (copies `framework/`, `installer/`, sets up minimal `.gitignore`), runs the installer against itself, asserts the managed block exists and contains every expected product, and that `git status` (if the fixture has a git repo) reports clean.
  - **Non-self-install regression**: the existing `install.bats` tests must continue passing without modification. Any test that asserts on `.gitignore` content must show that the managed block is NOT written when the consumer path differs from the Formann path.

- **Product enumeration (module 1):** not directly tested as a unit. Exercised through modules 2 and 3's tests, which depend on its correctness. Adding a fixture skill or agent to the synthetic checkout and asserting that the new path appears in the managed block is sufficient coverage.

- **Bootstrap and README issues:** not test-driven. Bootstrap is a one-off live operation verified by post-condition (`git status` clean, gitignore has the managed block). README is reviewed at triage/merge time.

- **Prior art:** `installer/tests/install.bats` is the existing test suite. New tests join it (or a sibling file if the file grows unwieldy). The `test_helper/` directory already has `bats-support` and `bats-assert` for assertions.

## Out of Scope

- **Symlink-graph divergence tooling.** A future contributor-tool that diffs the installed symlink graph against the enumerated products (to detect drift between installed state and current `framework/` contents) is not built here. The simpler answer — re-run the installer — is sufficient for v0.

- **Cross-machine bootstrap automation.** When the maintainer clones Formann on a new machine, running `./installer/install.sh .` once is the manual step. No script, alias, or CI hook is provided to automate this. Documented in the README.

- **Managed-block migration.** If the marker format ever changes, the installer does not migrate old marker pairs to new ones. The maintainer hand-edits during the marker-format change. Marker format is not expected to change frequently enough to warrant migration code.

- **Non-self-install gitignore tightening.** This PRD does not change what a regular consumer's `.gitignore` looks like beyond what the installer already writes (`/.formann`). The "consumers commit their stable symlinks" contract from ADR-0001 stands.

- **Honoring "do not edit" inside the managed block.** The installer will silently overwrite manual edits inside the block. No warning, no diff, no confirmation. If the maintainer hand-edits and re-runs the installer, the edits are lost. This is a documented limitation, not a future feature.

- **GitHub Issues binding's gitignore implications.** When a future binding lands, its products will be enumerated by the existing enumerator (which walks the chosen binding folder). No binding-specific gitignore logic is needed; the enumerator handles it generically.

- **Per-component opt-out.** The maintainer cannot tell the installer "ignore these products but track those." All installer products are in the managed block, all gitignored. If the maintainer wants to track a specific product, they hand-edit outside the markers — and accept that the installer's symlink for that path will continue to be regenerated. (In practice, this case has no real use.)

- **Renaming `installer/` to `framework/installer/` or similar.** The installer's location is unchanged. Any restructuring of where the installer lives is a separate concern.

## Further Notes

- The `.out-of-scope/per-issue-human-gate.md` file landed on Formann's `main` immediately before this PRD branch was cut. It is not part of this PRD's scope, but it is part of the broader dogfood narrative — Formann's `.out-of-scope/` convention is established by that commit, and this PRD's bootstrap issue completes the dogfood loop.

- The maintainer noted that a more elaborate dogfood-via-branch approach (a long-lived `dogfood` branch carrying installer products) was considered and rejected during the to-prd grilling. The argument: long-lived branches reintroduce the split-brain problem Formann is designed to avoid (per ADR-0001's single-source-of-truth principle). The managed-gitignore approach in this PRD preserves single-source-of-truth — `main` is what every consumer sees, the maintainer's machine just has additional per-machine state that any consumer also has after running the installer.

- The Formann `.gitignore` will, post-bootstrap, contain both the existing global ignores (`.DS_Store`, `__pycache__/`, etc.) and the managed block. A contributor-evaluator reading the file sees that Formann uses Formann to install itself — a self-evident dogfood signal.

- Issue 6 (Formann's own `.features/` tracker dogfood proof) is partly tautological: this PRD existing under `.features/self-install/PRD.md` is itself the proof. It exists as an explicit issue to capture the operational step of archiving the feature (`.features/self-install/` → `.features/.archived/self-install/`) once the other issues are done, which is the canonical end-of-feature ritual per the binding.
