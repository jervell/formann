The diff confirms both `--field query=` usages (mutation command and verification query command). Both use the wrong flag.

Let me also note one more thing to check: the verification query uses `--field id="$parent_node_id"`. In `gh api graphql`, when passing GraphQL variables, the correct approach is:
- For string variables that should not be type-coerced: `-f var=value` / `--raw-field var=value`
- For typed variables (numbers, booleans): `-F var=value` / `--field var=value`

For a GraphQL `ID!` type variable, using `-F`/`--field` is actually correct because the CLI sends the value as a typed field which becomes a string in the JSON variables object. So `--field parentId="$parent_node_id"` for ID variables is correct. But `--field query='...'` for the query string itself is wrong.

To summarize the `--field` issue: In `gh api graphql`:
- The `query` argument is a top-level string key → must use `-f`/`--raw-field`  
- GraphQL variable arguments like `parentId`, `subId`, `id` → `-F`/`--field` is acceptable

The BINDING.md uses `--field` for everything, including the query string. This is incorrect for the query parameter.

**Findings**

**Finding 1 — Wrong `gh api graphql` flag for the query string parameter**

Location: `/repo/framework/bindings/issue-tracker/github-issues/BINDING.md` lines 157 and 173

The two `gh api graphql` code blocks use `--field query='...'` to pass the GraphQL query string. In the GitHub CLI, `--field` is the long form of `-F` (typed field), which applies type coercion to the value. For a GraphQL query string containing curly braces and multi-line content, `-F`/`--field` will attempt to parse the value as typed JSON and fail. The correct flag for a literal string is `-f`/`--raw-field`.

Every other concrete usage of `gh api graphql` in this codebase uses `-f query=` for the query string:
- `/repo/framework/bindings/issue-tracker/github-issues/tracker-snapshot` line 113: `-f query="$_list_query"`
- `/repo/framework/bindings/issue-tracker/github-issues/tracker-snapshot` line 213: `-f query="$_slug_query"`

The PRD and the issue file itself also specify `-f query=`:
- `/repo/.features/github-issues-binding/PRD.md` line 119: `gh api graphql -f query='mutation { addSubIssue ... }'`
- `/repo/.features/github-issues-binding/issues/09-create-issue-verb.md` lines 22, 54, 63: same

The implementation comment's decision note compounds this by misidentifying the flag: "Used `--field` (long form) for `gh api graphql` arguments to be explicit about what `--field` / `-f` maps to." This is factually wrong — `--field` is the long form of `-F`, not `-f`. The long form of `-f` is `--raw-field`.

A skill author following the documented commands verbatim would get errors when running the mutation or verification query. The variable arguments (`--field parentId=`, `--field subId=`, `--field id=`) are correctly using `-F`/`--field` for typed GraphQL variable values; only the `query` parameter is wrong.

Severity: 🟡 Important — the commands as documented will not work, but this is a documentation issue with no runtime code impact; a maintainer following the docs will get a `gh` CLI error and need to correct the flag.
