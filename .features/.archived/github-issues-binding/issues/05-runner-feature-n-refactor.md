# Runner refactor: `(feature, N)` plumbing and `binding_native_ref` helper

> *This was generated by AI during to-issues.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

Refactor the runner so it never regex-parses `<feature>/NN` to derive the feature and never assumes the binding-native ref shape. This is binding-agnostic prep work: it lands before any GH-specific code and the local-markdown path stays identical end-to-end.

Three call sites consume the ref by regex-parsing today; each switches to consuming a `(feature, N)` pair plus, where needed, the binding-native ref form retrieved via a new helper:

1. **`dispatch_one`** — the snapshot's `feature` field and the issue's `nn` are both in scope. Pass them explicitly to downstream helpers instead of regex-parsing the ref. When the binding-native ref needs to be passed into a binding script, resolve it via `binding_native_ref`.
2. **`write_abort_flag`** — `.runner-state/aborted/<feature>/<NN>`. Continues to use `<feature>/<N>` as the filesystem path; both halves come from the `(feature, N)` pair, never from regex-parsing the binding-native ref.
3. **Classifier invocations** — wherever the runner passes the ref into the classifier, it now passes `(feature, N)` and resolves the binding-native ref via the helper if the classifier needs it.

New helper: `binding_native_ref <feature> <N>` looks up the binding-native ref by indexing into the current snapshot's issues array. The runner's CLI argument parser (`--issue <feature>/<N>`, regex `^([a-z0-9][a-z0-9-]*)/([0-9]+)$`) is unchanged — that's the **portable** form for operator ergonomics.

Also in this slice (binding-agnostic, related):

- Move HEAD-archive logic out of the `take_snapshot` wrapper and into `framework/bindings/issue-tracker/local-markdown/tracker-snapshot`. The wrapper shrinks to a thin shell-out; each binding owns its "current state of the world" definition.

Location: `/repo/framework/runner/run-the-queue.sh:362–363` (`format_summary_md`) and `/repo/framework/runner/run-the-queue.sh:434` (`format_multi_feature_summary_md`)

Problem: The SUMMARY.md log-link formatters extract the `nn` from the dispatch record's `ref` field by looking for a `/` character:

```awk
slash=index(ref, "/");
nn = (slash > 0) ? substr(ref, slash+1) : ref;
```

For a GitHub-issues binding ref `#42`, there is no `/`, so `nn` becomes `"#42"`. The generated link is `[#42.log](#42.log)`. But the actual log file created in `dispatch_one` is `42.log` (using the `nn` parameter passed to `dispatch_one`, not derived from the ref). The link is therefore broken for any GH-binding dispatch.

The same problem exists in `format_multi_feature_summary_md` (line 434), which uses `ref` directly for the log path: `[#42.log](#42.log)` instead of `[feature/42.log](feature/42.log)`.

No test covers GH-shaped refs in either formatter. This is not pre-existing — it becomes reachable for the first time because this refactor correctly routes binding-native refs (including `#42`) through `record_dispatch`.

Severity: 🟡 Important — GH-binding SUMMARY.md log links are incorrect. The files exist but are unreachable from the SUMMARY. This does not affect correctness of dispatch or propagation.

---

Location: `/repo/framework/runner/run-the-queue.sh:628` (`print_multi_feature_records`)

Problem: The drain-mode record association checks `[[ "$d_ref" == "$feature"/* ]]`. For a GH-binding ref like `#42`, this glob never matches a feature slug like `my-feature/*`. Dispatch records for GH-binding features in drain mode are silently dropped from the multi-feature SUMMARY — no `I|…` rows are emitted for GH-bound features.

This pattern was present before this commit (confirmed via git), so it is a pre-existing condition that this refactor brings into scope without fixing.

Severity: 🟡 Important — drain-mode SUMMARY for GH-binding is incomplete (no per-issue rows for GH features). However, this is pre-existing and the issue explicitly calls this slice "binding-agnostic prep work" with GH-specific work deferred. Whether this gap needs to be tracked as a follow-up is a maintainer decision.
