# Fix same-day same-kind comment suffix algorithm in BINDING.md

> *This was generated by AI during to-issues.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

The recipe at `framework/bindings/issue-tracker/github-issues/BINDING.md:234-241` documents how to determine the `(N)` suffix on same-day same-kind comments:

```
gh issue view <N> --json comments \
  --jq '[.comments[].body | split("\n")[0]] | map(select(startswith("### <Kind> — <YYYY-MM-DD>"))) | length'
```

This `startswith` check undercounts past the first collision: once `### Implementation (2) — 2026-05-16` exists, it no longer starts with the un-suffixed prefix `### Implementation — 2026-05-16` (character 17 is `(` vs `—`), so the third call sees `count == 1` and posts another `(2)` — producing a duplicate suffix. The bug recurs every third same-day same-kind comment.

Replace `startswith` with a pattern/regex match that counts both un-suffixed and `(N)`-suffixed prior headings. For example:

```
[.comments[].body | split("\n")[0]] |
  map(select(test("^### <Kind>( \\([0-9]+\\))? — <YYYY-MM-DD>$"))) | length
```

Update the BINDING.md recipe accordingly and add an example walking through three same-day Implementation comments to show the suffix progression `unsuffixed → (2) → (3)`.

## Findings

### Bug-hunt

**No bugs found.** I traced the filter in full detail, byte by byte.

The two backslash characters before each paren in the file (`\\(` and `\\)`) are correctly handled:

- The `--jq` argument uses bash single-quotes, so no shell transformation occurs.
- jq receives the literal bytes `\\` then `(`. In jq's JSON string parser, `\\` is the escape sequence for a single literal backslash. The following `(` is then just a literal character — it does not trigger jq's `\(` string-interpolation mechanism, because the preceding `\\` was already fully consumed as an escape.
- The jq string value is therefore `\(` and `\)` — which in POSIX ERE (Oniguruma) match a literal open-paren and close-paren respectively.
- The `$` at the end of the regex is not escaped in the jq string (`$` has no special meaning in jq string literals), so it reaches the regex engine as a literal `$`, which is the end-of-string anchor. Correct.

The outer ERE group `( \([0-9]+\))?` correctly matches the optional ` (N)` suffix. I tested all key cases (unsuffixed, `(2)`, `(3)`, wrong date, wrong kind, trailing garbage, empty suffix) with an equivalent Python regex and all produced the expected results.

### Intent-check

The original bug was: the third same-day same-kind call saw `count=1` (only the unsuffixed heading matched) and posted a duplicate `(2)`. The fix correctly counts both `### Implementation — 2026-05-16` and `### Implementation (2) — 2026-05-16`, returning `count=2` on the third call and posting `(3)`. The worked example in the table accurately reflects this.

The choice of hybrid approach over pure regex is well-reasoned. The `Review (AFK gate)` kind is documented in the binding and contains unescaped parentheses that would require per-kind escaping in a pure regex form. The hybrid avoids that entirely.
