# `sandbox-env` runner hook + GH binding's Keychain emitter

> *This was generated by AI during to-issues.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

Two pieces ship together so the runner stays binding-agnostic while the GH binding can declare its sandbox prerequisites:

1. **Binding-agnostic runner hook.** Before `docker run`, the runner checks whether `$HOST_REPO/docs/formann/issue-tracker/sandbox-env` (a path on the role surface) is executable. If yes, invoke it and append its stdout — one `KEY=value` line per env var — to the container's `--env-file`. local-markdown ships no script; the hook is a no-op there.

2. **GH binding's `sandbox-env` script.** Ships at `framework/bindings/issue-tracker/github-issues/sandbox-env`. Retrieves the GH token from macOS Keychain (mirroring `retrieve-token.sh`'s pattern) and emits `GH_TOKEN=<token>` on stdout. Exits non-zero with a clear stderr error when the Keychain entry is absent.

This keeps `run-the-queue.sh` binding-agnostic by construction — the runner never imports `gh`, `GH_TOKEN`, or any GitHub knowledge. The binding declares its own sandbox needs.

**Summary:** Add a binding-agnostic `sandbox-env` hook before `docker run` and ship the GH binding's `sandbox-env` script that emits `GH_TOKEN` from macOS Keychain. The runner stays free of any GitHub-specific knowledge.

**Current behavior:**
`framework/runner/run-the-queue.sh` constructs the `docker run` invocation with `--env-file <(printf 'CLAUDE_CODE_OAUTH_TOKEN=%s\n' "$TOKEN") ...` (the Claude OAuth token is the only env var plumbed in). There is no hook for a binding to declare additional env vars. The runner has no way to learn about `GH_TOKEN` without importing GitHub-specific code, which violates the binding-agnostic-runner contract (ADR-0004).

**Desired behavior:**

Two pieces ship together:

1. **Runner-side: binding-agnostic `sandbox-env` hook.** Before `docker run`, the runner checks whether `$HOST_REPO/docs/formann/issue-tracker/sandbox-env` (a path on the role surface) is executable. If yes, invoke it and append its stdout — one `KEY=value` line per env var — to the container's env. local-markdown ships no script; the hook is a no-op there. Malformed lines (empty, lines without `=`, lines with shell metacharacters that could escape into the env-file) are either skipped with a stderr warning or rejected with a clear error; the decision is documented in code comments at the hook.

2. **GH binding: `sandbox-env` emitter.** `framework/bindings/issue-tracker/github-issues/sandbox-env` exists, is executable, retrieves the GH token from macOS Keychain (mirroring `framework/runner/retrieve-token.sh`'s pattern), and emits `GH_TOKEN=<token>` on stdout. Missing Keychain entry → exit non-zero with a clear stderr error.

`run-the-queue.sh` contains zero GitHub-specific knowledge (no `gh`, no `GH_TOKEN`, no Keychain code). BINDING.md's "Sandbox compatibility" section documents `GH_TOKEN` as the required env var and `api.github.com` as the egress requirement.

**Key interfaces:**
- Hook path: `$HOST_REPO/docs/formann/issue-tracker/sandbox-env` — the role-surface entry point. The role-surface symlink (slice 13) points at the chosen binding's folder; the script (if any) lands at the binding's folder root.
- Existing docker run env plumbing in `run-the-queue.sh` uses `--env-file <(printf ...)`. The hook's stdout extends that env-file content; the hook is invoked before the `docker run` line and its output is appended to the env-file.
- Hook malformed-line policy: documented decision in code comments at the call site. Defensive parsing (anchored regex `^[A-Z_][A-Z0-9_]*=` for the key, then everything after `=` as value; reject anything that doesn't match).
- `framework/runner/retrieve-token.sh` is the reference pattern for Keychain retrieval (uses macOS `security find-generic-password`). The GH binding's `sandbox-env` follows the same shape.
- BINDING.md's "Sandbox compatibility" section documents `GH_TOKEN` + `api.github.com` egress as the binding's sandbox prerequisites.
