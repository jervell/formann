## What to build

Ship the GH binding's `tracker-snapshot` script at `framework/bindings/issue-tracker/github-issues/tracker-snapshot`. Schema-compatible with local-markdown's JSON contract (same keys and shape; the `ref` field's value format differs — `#N` here, `<feature>/NN` for local-markdown).

Two invocation forms:

- `tracker-snapshot --list` — returns open `formann:feature`-labelled parent issues, ordered by issue number ascending (binding-specific order; documented in BINDING.md).
- `tracker-snapshot <slug>` — returns the snapshot for one feature: the parent, all its sub-issues, their bodies, and their labels. One GraphQL query per invocation.

Behaviors to implement:

- Single GraphQL query fetches the parent, sub-issues, bodies, labels in one round-trip.
- Sub-issue ordering follows the GraphQL `subIssues` connection's priority order (UI-mutable via drag-and-drop), with `#N` ascending as the tiebreaker.
- Blocker eligibility resolves from the in-memory sub-issue snapshot — no extra API calls. `#N` refs in `## Blocked by` are matched against the parent's sub-issue set; unmatched refs are conservative-false on eligibility (matching local-markdown's cross-feature behaviour).
- Archive filtering: `--list` queries `is:open label:formann:feature`, so closed (archived) parents drop out naturally.
- Slug-uniqueness cardinality cases:
  - 0 matches → emit `{"feature":"<slug>","issues":[]}`, exit 0 (mirrors local-markdown's missing-feature-dir behaviour).
  - 1 match → normal snapshot.
  - ≥2 matches → exit non-zero with stderr naming both parent issues and the recovery recipe ("remove the `formann:slug:<slug>` label from one of: #N, #M").
- Transient API failures (rate-limited, 5xx, auth-expired) surface with clear exit codes after a bounded retry. Maintainer can distinguish "binding transient failure" from "issue malformed".
- Offline-testable via a `gh` shim on `PATH` reading recorded GraphQL responses from a fixture directory.

## Findings

### 1. The `#N` ascending tiebreaker is unimplemented

**Location:** `tracker-snapshot`, lines 232–236 (comment), and the entire Phase 1/Phase 2 block.

**Problem:** The acceptance criteria state "`<slug>` mode orders sub-issues by the `subIssues` connection's priority order with `#N` ascending as tiebreaker." BINDING.md repeats this contract ("Sub-issues are emitted in the order returned by the GraphQL `subIssues` connection... `#N` ascending is applied as a stable tiebreaker within equal-priority groups."). The code does neither: it reads sub-issues via `jq -c '.data.repository.issues.nodes[0].subIssues.nodes[]'` and appends them to parallel arrays in encounter order. There is no `sort -n` call anywhere in the script. The comment at line 234 is self-contradictory — it says the tiebreaker is "applied via sort -n on a secondary key" in one clause, then says "since each position is unique it never changes the primary ordering" in the next, describing a sort that has no effect and is never actually in the code.

The "multi-issue-priority-tiebreaker" test fixture returns issues in API order `[#10, #7, #15]` — a non-ascending sequence — and the test simply verifies that order is preserved end-to-end. It does not exercise an equal-priority group where `#N` ascending would matter. So this scenario is untested as well as unimplemented.

The evidence block claims this criterion is satisfied and cites the test, but the test demonstrates API-order passthrough, not `#N` tiebreaking.

**Severity:** 🟡 Important — the stated contract (BINDING.md, AC, implementation notes) is not met. The practical impact is non-deterministic ordering of equal-priority sub-issues when the GitHub API returns them in an arbitrary order; the AFK runner could pick a different issue than intended. The comment also actively misleads anyone reading the code.

The test name promises that a closed parent is absent from the result; the fixture provides no closed parent to exclude.

This means the "archived (closed) parent excluded from --list" fixture scenario required by the AC is not actually a scenario test — it is an open-only baseline test that happens to be named after an exclusion property it cannot verify.

**Severity:** 🟡 Important — this is an evidence-check mismatch. The evidence says this scenario is covered, but a closed-parent entry would need to appear in the fixture and be absent from the output for the test to prove the claim.

### 4. Zero-match path emits slug without JSON escaping before feeding to `jq`

**Location:** `tracker-snapshot`, line 220.

```bash
printf '{"feature":"%s","issues":[]}\n' "$_slug" | jq -c '.'
```

**Problem:** The slug is interpolated directly into the JSON string before piping to `jq -c '.'`. If `$_slug` contains a double-quote or backslash (possible, since `$1` is user-controlled), `printf` will produce syntactically malformed JSON. The `jq -c '.'` call would then fail to parse it, emitting nothing to stdout. With `set -e` / `set -o pipefail` active, the script exits non-zero. The 1-match path (line 322) correctly uses `$(printf '%s' "$_slug" | jq -Rs '.')` which performs proper JSON encoding.

This is exploitable only with a slug that contains `"` or `\` — unusual in practice since slugs come from GitHub label values that are unlikely to have these characters — but the two code paths are inconsistent, and the 0-match path would fail silently rather than emitting a valid empty snapshot.

**Severity:** 🟢 Minor — real-world slugs follow a `[a-z0-9-]+` pattern and this cannot be triggered with a valid label name; but the inconsistency with line 322 and the silent failure mode on malformed input is worth noting.
