# Drop the post-gate dirty-checkout override ‚Äî the classifier verdict is the truth

> *This was generated by AI during incident triage of the 2026-05-18 runner pass.*

## Parent

`.features/github-issues-binding/PRD.md`

## What to build

`classify_gate_outcome` (`framework/runner/run-the-queue.sh:130-158`) reduces `(gate_rc, pre/post snapshot delta)` to a verdict ‚Äî `clean | blocked | gate-failed` ‚Äî built from **committed state only** (`tracker-snapshot` reads HEAD via `git archive`, not the working tree). That verdict is the truth about what the gate did. It is the signal the operator cares about.

The post-gate dirty-checkout override at `run-the-queue.sh:1623-1625` discards that verdict whenever `git status --porcelain` is non-empty and forces `gate-failed`. The `gate-failed` branch at line 1649-1654 then refuses to propagate the gate's committed tracker work to host and writes an abort flag. So any incidental working-tree leftover ‚Äî a kernel core dump from a crashed in-container subprocess (`core_pattern=core`, CWD-relative), a stray write ‚Äî strands a legitimate `blocked` or `clean` verdict and silently buries the gate's findings.

Concretely: in `.runner-state/runs/20260518-080643/`, the gate for issue 21 ran cleanly, identified real bugs in the implement step's fix, and committed `fd52c31 tracker: review github-issues-binding/21 ‚Üí blocked` in the runner-checkout. The implement step had leaked a 0-byte `framework/runner/tests/core` file (subprocess crash); that leftover persisted into the post-gate working tree, the override forced `gate-failed`, and the blocked-verdict commit was stranded ‚Äî host got no record of the review's findings until the maintainer cherry-picked manually.

The fix is to delete the override. The classifier verdict propagates unchanged. The redundant post-gate dirty-warning at line 1644-1647 ‚Äî which re-prints the same lines the implement-stage warning at line 1524-1527 already printed ‚Äî also goes. If the operator wants more than the verdict, the gate's log file and the issue's comment trail are right there.

## Agent Brief

> *This was generated by AI during incident triage.*

**Summary:** The classifier verdict from `classify_gate_outcome` is the truth about what the gate did. The post-gate dirty-checkout override discards it on incidental working-tree leftovers, stranding legitimate `blocked` and `clean` verdicts. Delete the override, delete the redundant re-warning, and stop reading the post-gate working tree at all.

**Current behavior:**

- `dispatch_one` runs the gate via `run_gate_container` and captures `gate_rc`.
- Reads `gate_dirty = git status --porcelain` at line 1597-1598.
- `classify_gate_outcome` produces a verdict from `(gate_rc, pre/post snapshot delta)`. The snapshot reads HEAD (`git archive`), so the verdict is committed-state truth.
- The override at line 1623-1625 forces `gate_verdict = "gate-failed"` whenever `gate_dirty` is non-empty.
- The `gate-failed` branch at line 1649 writes an abort flag and refuses to propagate the gate's committed tracker work to host.
- Line 1644-1647 re-prints `gate_dirty` ‚Äî identical content to what the implement-stage warning at line 1524 already printed when the leak happened.

**Desired behavior:**

- The classifier verdict propagates unchanged: `clean` ‚Üí status flips to `done` on host; `blocked` ‚Üí gate's tracker comment reaches host, issue held at `in-review` for maintainer review; `gate-failed` ‚Üí abort flag, no propagation.
- The runner does not read the post-gate working tree at all. There is no override and no re-warning.
- The implement-stage warning is the single point of leak-reporting per iteration. Anyone who wants to dig further has the gate's log and the issue's commit/comment trail.
- The next iteration's `ensure_runner_checkout` cleans any working-tree leftovers via `git clean -fd` before the next dispatch ‚Äî that's the existing, tested cleanup path.

**Why "trust the classifier" is safe:**

`tracker-snapshot` reads `git archive HEAD .features` when invoked by the runner (`framework/bindings/issue-tracker/local-markdown/tracker-snapshot:43-49`; the github-issues binding queries the live API, which is also committed-state). The classifier therefore sees only what the gate **committed**. An off-mission gate that writes the tracker file but does not commit produces no snapshot delta, and the classifier returns `blocked` ‚Äî the issue is correctly held for the maintainer. The override added no behavioral correctness on top of that; it just relabeled `blocked` as `gate-failed` and refused to propagate work that didn't exist to propagate. On the legitimate paths (gate committed real tracker work AND incidental working-tree leftover existed), the override actively destroyed value.

**Symmetry with the implement stage:**

The implement-stage code already trusts its classifier (`classify_outcome`) and only warns about uncommitted state. It does not override the classifier on dirt. After this change, the gate stage matches that pattern.

**Key interfaces:**

- `framework/runner/run-the-queue.sh:1597-1598` ‚Äî `gate_dirty` capture. Delete.
- `framework/runner/run-the-queue.sh:1619-1626` ‚Äî override block (4 lines of comment + 3-line `if/then/fi`). Delete.
- `framework/runner/run-the-queue.sh:1644-1647` ‚Äî redundant post-gate dirty re-print. Delete.
- `framework/runner/tests/run-the-queue.bats:1604-1639` ‚Äî transform the existing override-pinning test into the new-contract test per the AC list above. No other test files need touching; classifier-level coverage already exists at `bats:108-113`.

**Out of scope:**

- Detecting gate sessions that go off-mission by writing files outside the tracker. The classifier already handles this correctly: no snapshot delta ‚Üí `blocked` ‚Üí issue held. The override added no additional detection.
- Detecting gate sessions that commit off-mission content (e.g. mutating non-tracker files in the same commit). The override never caught this case either ‚Äî committed off-mission content leaves the working tree clean. After this change, the gate's commit-shape is constrained only by the gate prompt (`framework/runner/review-and-gate.md`), which is honor-system. A commit-shape validator (single-file, tracker path only) would be a separate, additive change; worth filing as its own issue if the honor-system gap matters.
- Preventing core dumps in the dispatch container. The kernel writes them on subprocess crash with default `core_pattern=core`; the iteration-start `git clean -fd` is the right cleanup mechanism.
- Reworking `classify_gate_outcome`. The classifier is correct.
