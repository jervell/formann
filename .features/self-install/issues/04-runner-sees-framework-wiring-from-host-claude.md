# Runner sees framework wiring from host `.claude/`

> *This was generated by AI during to-issues.*

## Parent

`.features/self-install/PRD.md`

## What to build

Mount host's `.claude/skills/`, `.claude/agents/`, and `.claude/rules/` (when present) read-only into the AFK runner's dispatch container, mirroring the existing `.formann` mount's shape. Without this, runner-checkouts produced by `git clone $HOST_REPO` lack the framework's `.claude/*` symlinks in self-install mode (where they're gitignored per `self-install/01`), and `claude -p` inside the container reports `Unknown command: /implement` on dispatch.

For normal consumers (iot today), this mount is behaviorally a no-op: host's `.claude/skills/*` and `.claude/agents/*` are tracked symlinks already present in the runner-checkout, and the overlay mounts identical content. For Formann self-install, the mount is what makes AFK dispatch possible at all.

## Acceptance criteria

- [ ] `run-the-queue.sh`'s docker invocation adds a `-v` read-only mount for `$HOST_REPO/.claude/skills` at the container's `.claude/skills` path when the host directory exists
- [ ] Same for `$HOST_REPO/.claude/agents`
- [ ] Same for `$HOST_REPO/.claude/rules` — skip when absent (mirrors the existing `link_rules` "skip if absent" pattern in the installer)
- [ ] All three mounts use `:ro`, consistent with the existing `.formann` mount
- [ ] No `.claude/` subdirectory beyond `skills`, `agents`, `rules` is mounted (no `settings.local.json`, `plugins/`, `worktrees/`, `plans/`, `scripts/`, `docs/`)
- [ ] When host's directory is missing or empty, the runner does not error — the corresponding mount is simply omitted
- [ ] Existing `installer/tests/install.bats` and any `framework/runner/tests/` tests pass unmodified
- [ ] Dispatch-shape coverage (new or extended bats test) asserts the docker invocation includes the expected `.claude/{skills,agents,rules}` mounts when those directories exist on host
- [ ] In a synthetic self-install dispatch (host has no tracked `.claude/*` symlinks), the container's Claude Code discovers framework skills — verifiable by a successful `/implement` invocation or `claude` listing framework skills [human]

## Agent Brief

> *This was generated by AI during triage.*

**Summary:** Add three read-only docker volume mounts to the AFK runner's dispatch invocation so that framework skill discovery works inside the container regardless of whether the host's `.claude/*` symlinks are tracked or gitignored.

**Current behavior:**
`framework/runner/run-the-queue.sh`'s dispatch step (`docker run --rm -t ...`, around the body of the function that builds the container args) mounts two host paths:
- `$HOST_CHECKOUT:$RUNNER_CONTAINER_REPO_PATH` — the cloned runner-checkout as the container's workdir
- `$HOST_REPO/.formann:$RUNNER_CONTAINER_REPO_PATH/.formann:ro` — host's framework checkout via the `.formann` indirection

The comment at this site explains the design assumption: `.claude/skills/<name>` symlinks resolve through `.formann`, and the mount handles the `.formann` half because `.formann` is untracked. The implicit assumption is that the `.claude/skills/<name>` symlinks themselves are tracked in the host repo and therefore present in the runner-checkout after `git clone`. For iot this holds — `git ls-files .claude/skills/triage` returns the symlink. For Formann in self-install mode (per `self-install/01`), the same symlinks are gitignored, so `git clone` does not include them, and the container sees no skills under `.claude/`.

**Desired behavior:**
The dispatch step adds three additional read-only mounts, each conditional on the corresponding host directory existing:

- `$HOST_REPO/.claude/skills:$RUNNER_CONTAINER_REPO_PATH/.claude/skills:ro`
- `$HOST_REPO/.claude/agents:$RUNNER_CONTAINER_REPO_PATH/.claude/agents:ro`
- `$HOST_REPO/.claude/rules:$RUNNER_CONTAINER_REPO_PATH/.claude/rules:ro`

If a given source directory does not exist on the host (e.g. `framework/rules` is empty and the installer never created `.claude/rules`), the corresponding `-v` flag is omitted entirely — docker would create an empty directory at the mount point otherwise, which is harmless but noisy.

The mounts overlay whatever the runner-checkout already has at those paths. For iot, the overlay is byte-equivalent to the cloned content (same symlinks resolving through the same `.formann` mount). For Formann self-install, the overlay populates paths the runner-checkout left empty.

No other `.claude/` content is mounted. `settings.local.json`, `plugins/`, `worktrees/`, `plans/`, `scripts/`, `docs/` stay on the host — they contain local Claude Code state the container has no business with, per the side-effects analysis that produced this issue.

**Key interfaces:**
- The dispatch function in `framework/runner/run-the-queue.sh` that constructs and invokes `docker run` for `/implement`. The same function (or a sibling) handles the review-and-gate dispatch — both invocations need the mounts.
- `HOST_REPO` is already resolved by `resolve_host_repo()` and available as a global at dispatch time; reuse it.
- The conditional pattern: bash `[ -d "$HOST_REPO/.claude/skills" ]` check guarding each `-v` arg's emission, similar in shape to the `[ -d "$source_dir" ]` guard the installer's `link_rules` uses.
- The container path prefix is `$RUNNER_CONTAINER_REPO_PATH` (existing variable); the suffix is `/.claude/<subdir>`.

**Acceptance criteria:**
- [ ] `run-the-queue.sh`'s `/implement`-dispatch docker invocation includes `-v "$HOST_REPO/.claude/skills:$RUNNER_CONTAINER_REPO_PATH/.claude/skills:ro"` when `$HOST_REPO/.claude/skills` exists
- [ ] Same for `.claude/agents`
- [ ] Same for `.claude/rules` — but the entire `-v` arg is omitted when the host directory does not exist
- [ ] The same three mounts are added to the review-and-gate dispatch invocation (both `docker run` sites need the wiring; tests should cover both)
- [ ] All mounts are `:ro`
- [ ] When a host directory exists but is empty, the mount is added and the container sees an empty directory (no special handling)
- [ ] No additional `.claude/` paths are mounted — `settings.local.json`, `plugins/`, `worktrees/`, `plans/`, `scripts/`, `docs/` remain on host only
- [ ] Existing tests pass unmodified
- [ ] New or extended bats coverage asserts the docker arg list contains the expected `-v` flags when the host directories exist, and omits them when they don't
- [ ] End-to-end verification on a self-install host: `claude` in the container discovers framework skills under `.claude/skills/`, and a `/implement` dispatch on a fixture issue completes the expected status transition [human]

**Out of scope:**
- Mounting any `.claude/` content beyond `skills`, `agents`, `rules` (decision made in the side-effects analysis; revisit if a new framework artifact category lands under `.claude/`)
- Self-install detection in the runner — the mounts are gated only on directory existence, not on whether the host is a Formann self-install vs a normal consumer
- Read-write mounts — `:ro` is the contract; containers do not mutate host `.claude/`
- Refactoring the existing `.formann` mount or the docker invocation's general shape — minimal-change addition
- Container-side skill resolution validation — that's `claude`'s concern, not the runner's
