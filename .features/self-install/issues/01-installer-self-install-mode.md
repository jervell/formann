# Installer self-install mode

> *This was generated by AI during to-issues.*

## Parent

`.features/self-install/PRD.md`

## What to build

When `installer/install.sh` is invoked with a consumer path that resolves to the Formann checkout itself, write a delimited managed block in the consumer's `.gitignore` listing every installer product (including `/.formann`). Outside-block content is preserved across runs. On re-run, the block is regenerated from current `framework/` contents so that adding a skill/agent/rule and re-running keeps the ignore list in sync. A regular consumer install (consumer path ≠ Formann path) continues to write only `/.formann` and no managed block — the existing contract from ADR-0001 is unchanged.

Three internal pieces ship together as one slice because none delivers observable value alone: (1) an enumerator function returning the list of paths the installer would create for the current Formann checkout and chosen role bindings; (2) a managed-block writer that finds or creates the marker pair in `.gitignore` and replaces the content between them with the enumerated list; (3) a self-install detection check (`-ef` on canonical paths) that gates the writer's mode.

## Acceptance criteria

- [ ] Installer detects self-install via canonical-path equivalence on the resolved consumer and Formann paths
- [ ] Self-install writes a managed block delimited by `# === Formann self-install (managed by installer — do not edit) ===` (start) and `# === /Formann self-install ===` (end)
- [ ] Managed block content includes `/.formann` plus every per-component product: skills (`/.claude/skills/<name>`), agents (`/.claude/agents/<name>.md`), rules (`/.claude/rules/<name>` when present), the chosen binding folders (`/docs/formann/<role>`), the flat role docs (`/docs/formann/<doc>.md`), and `/runner/Dockerfile`
- [ ] Re-running the installer regenerates the block from current `framework/` contents (adding `framework/skills/foo` and re-running adds `/.claude/skills/foo` to the block)
- [ ] Content outside the markers is preserved across runs
- [ ] Manual edits inside the markers are silently overwritten on next run (documented limitation, no warning emitted)
- [ ] Consumer install (consumer path ≠ Formann path) writes only `/.formann`, no managed block
- [ ] Existing `installer/tests/install.bats` tests pass unmodified
- [ ] New bats tests cover the managed-block writer against fixture `.gitignore` inputs: empty, with unrelated entries, with existing block, with stale block (extra/missing entries), with manually-edited block
- [ ] New bats test covers end-to-end self-install in a fixture Formann checkout: assert managed block present and complete after install
- [ ] No new shell runtime dependencies introduced (pure bash; same constructs as today: globs, arrays, parameter expansion, `printf`/`echo`, `grep`/`sed`)

## Agent Brief

> *This was generated by AI during triage.*

**Summary:** Teach `installer/install.sh` to detect self-install (consumer path resolves to Formann path) and, in that mode only, manage a delimited block in `.gitignore` listing every installer product.

**Current behavior:**
`installer/install.sh`'s `update_gitignore` appends `/.formann` once (idempotent) to the consumer's `.gitignore` and writes nothing else. Other installer products — `.claude/skills/<name>`, `.claude/agents/<name>.md`, `.claude/rules/<name>` (when present), `docs/formann/<role>` binding folders, `docs/formann/<doc>.md` flat role docs, and `runner/Dockerfile` — are committed by the consumer per ADR-0001, since their relative paths are stable across machines via the `.formann` indirection. Each `link_*` function enumerates its source directory inline (`for entry in "$source_dir"/*/`, etc.) — the enumeration is duplicated across functions.

When the consumer path equals the Formann path (the maintainer running the installer on Formann itself), the symlinks become one-level-of-indirection redirects to canonical `framework/` content, and committing them would put dangling-on-fresh-clone symlinks on `main` for consumer-evaluators. Not committing them leaves `git status` permanently noisy.

**Desired behavior:**
- After path resolution in `main()`, the installer checks whether the resolved consumer path and Formann path are equivalent (`[ "$consumer_path" -ef "$formann_path" ]`) and sets a self-install flag.
- `update_gitignore` becomes a two-mode function:
  - **Normal mode** (flag unset): unchanged — append `/.formann` if missing.
  - **Self-install mode** (flag set): write/rewrite a managed block delimited by `# === Formann self-install (managed by installer — do not edit) ===` (start) and `# === /Formann self-install ===` (end). The block lists every installer product (one per line, each prefixed with `/` to anchor at consumer root): `/.formann` first, then per-skill `/.claude/skills/<name>`, per-agent `/.claude/agents/<name>.md`, per-rule `/.claude/rules/<name>`, per-role-binding `/docs/formann/<role>`, per-flat-doc `/docs/formann/<doc>.md`, and `/runner/Dockerfile`.
- Content outside the marker pair is preserved verbatim across runs. Content inside is regenerated from scratch from the current framework contents.
- Manual edits inside the markers are silently overwritten on the next install run — no warning, no diff, no confirmation. This is a documented limitation, not a feature.

**Key interfaces:**
- **Product enumeration function** — returns the relative paths (from consumer root, leading `/` for gitignore anchoring) of every artifact the installer would create for the current Formann checkout and the chosen role bindings. Walks `framework/skills/*/` (per-folder), `framework/agents/*.md` (per-file), `framework/rules/*` (per-file when present), the chosen binding folders (one per role), the flat role docs (lifecycle.md, inbox.md, domain.md, triage-states.md, afk-runner.md, afk-runner-flow.md — same list `link_role_surface` uses today), and `runner/Dockerfile`. The existing `link_skills`, `link_agents`, `link_rules`, and `link_role_surface` functions should be refactored to call into this enumerator so there is one source of truth for "what does the installer produce." Inline traversals in `link_*` go away.
- **Managed-block writer function** — takes the enumerated list, reads `.gitignore`, finds or creates the marker pair, replaces content between markers with the list (one path per line). Preserves outside-marker content. Idempotent: running twice with the same input produces the same output.
- **Self-install detection** — `[ "$consumer_path" -ef "$formann_path" ]` after both paths are resolved via the existing `cd ... && pwd` pattern. The check handles trailing slashes, symlinks, and relative paths uniformly.
- **`update_gitignore` dispatcher** — reads the self-install flag and routes to either the legacy single-line append or the managed-block writer.

**Acceptance criteria:**
- [ ] Self-install detection uses `-ef` (or equivalent) on the resolved consumer and Formann paths
- [ ] In self-install mode, `.gitignore` contains a managed block delimited by the specified marker pair
- [ ] The managed block lists `/.formann` plus every installer product: skills, agents, rules (when present), role-surface binding folders, flat role docs, and `/runner/Dockerfile`
- [ ] Adding a skill folder under `framework/skills/` and re-running the installer adds the corresponding `/.claude/skills/<name>` entry to the block
- [ ] Content outside the marker pair is preserved across runs
- [ ] Manual edits inside the marker pair are silently overwritten on next run (no warning emitted)
- [ ] In normal-consumer mode (consumer path ≠ Formann path), `.gitignore` receives only `/.formann` (idempotent append) and no managed block
- [ ] The existing `link_*` functions are refactored to consume the enumerator function (no duplicate enumeration logic remains)
- [ ] Existing `installer/tests/install.bats` tests pass unmodified
- [ ] New bats tests cover the managed-block writer against fixture `.gitignore` inputs: empty, with unrelated entries, with existing valid block, with stale block (missing/extra entries), with manually-edited block (assert overwritten)
- [ ] New bats test covers self-install end-to-end: set both `FORMANN_PATH` and consumer path to the same directory, run the installer, assert the managed block contains every expected product
- [ ] No new shell runtime dependencies introduced (pure bash; same constructs as today)

**Out of scope:**
- Warning the maintainer when manual edits inside the managed block are about to be overwritten (silent overwrite is the documented behavior)
- Symlink-graph divergence detection (re-run the installer when in doubt)
- Marker-format migration code (if the marker pair changes, hand-edit)
- Any change to normal-consumer `.gitignore` behavior — only `/.formann`, unchanged
- Adding a `--non-interactive` flag or other CLI ergonomics — outside this slice

**What changed**

- `installer/install.sh`: added `enumerate_products` as the single source of truth for what the installer creates. Each line is `<kind>\t<dest>\t<target>` with `kind` ∈ {`symlink`, `copy`}. `install_products` consumes that stream and realises each product. `is_self_install` is a one-liner over `-ef`. `write_managed_block` does an in-place rewrite when the marker pair exists (preserving the block's position in the file and all content outside it) or appends a fresh block at end of file otherwise. `update_gitignore` dispatches by the self-install flag — consumer mode is the legacy idempotent `/.formann` append.
- `installer/tests/self-install.bats`: new sibling test file. Setup builds a synthetic Formann checkout under `$BATS_TEST_TMPDIR/formann-self` and runs the installer with `FORMANN_PATH` and the consumer argument both pointing at that fixture.

**Decisions made**

- The brief said "the existing `link_*` functions should be refactored to call into this enumerator". I removed those wrapper names entirely rather than keep them as one-line filter wrappers — the single `install_products` reader is the cleanest expression of "no duplicate enumeration logic" and the legacy names had no remaining behavioural distinction.
- `enumerate_products` is captured once in `main` (`products="$(enumerate_products …)"`) and piped into both `install_products` and `write_managed_block`. This keeps the "skipping skills / agents / rules" stderr warnings (already asserted on by `installer/tests/install.bats`) emitting exactly once per run, even in self-install mode where both consumers fire.
- The writer uses `mktemp` for the in-place rewrite tmpfile and `tail -c1` to decide whether the existing file needs a trailing newline before a fresh block is appended. Both are POSIX core utilities; no new languages or heavyweight deps were introduced.
- The writer rewrites in-place when the marker pair exists, preserving the block's position in the `.gitignore` (so above/below content stays exactly where it was). The append branch only fires when the pair is absent.
